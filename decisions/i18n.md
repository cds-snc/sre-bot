# Internationalization

## Context

English and French are peers — an Official Languages obligation, not a nice-to-have. Two systems currently coexist: legacy modules use `python-i18n` against central `app/locales/*.yml`; new features use the in-house `app/infrastructure/i18n/` stack (YAML catalogues per feature, resource registry, hookspec registration). The old Draft record proposed selecting an external library and was overtaken by the code; catalogue migration to gettext would be a format, tooling, and workflow change with no proportional payoff for short chat strings.

## Decision

**The in-house `infrastructure/i18n` stack is the adopted system.** YAML key catalogues, per-feature `locales/` directories registered via the `register_i18n_resources` hookspec, an `I18nService` consumed by transports and features. Rationale over gettext/Babel/Fluent: our strings are short, parameterized chat/API messages authored by developers, not documents routed to external translators; YAML-per-feature keeps translations in the vertical slice. Revisit only if professional translation workflows become a requirement.

**The bilingualism guarantee is a CI gate, not a runtime fallback:** every user-facing key must exist in both `en` and `fr` catalogues — the **parity check** fails the build otherwise. Runtime fallback to the default locale still exists, but only as a safety net for deploy-time races; it is not the compliance mechanism, because "both languages always available" cannot be enforced by a fallback that papers over the missing one.

**Locale resolution:** per-request, from the platform signal (Slack payload locale, `Accept-Language`), bound to `contextvars` at the inbound boundary alongside correlation. Services and renderers read it from context; passing `locale` explicitly is fine in pure helpers — pick per function, don't thread both.

**Slack help/commands:** localized via `description_key`/`example_keys` referencing catalogue keys; plain-literal `description`/`examples` are deprecated once parity covers the registered commands ([transport-slack.md](transport-slack.md)).

**Legacy:** modules on `python-i18n` migrate to the stack as each module goes through [migration.md](migration.md); `python-i18n` leaves the dependency tree when the last one lands.

## Consequences

- Owning the stack means owning plural rules and formatting edge cases; accepted for chat-sized strings.
- The parity gate is the whole point: missing-French becomes a build failure, not a production surprise.

## Checks

- CI parity check across every registered catalogue (en ↔ fr key sets equal).
- New feature PRs with user-facing text include both catalogues.
- grep (post-migration): no `import i18n` (python-i18n) outside `app/modules/`.

## Migration

Ticket: parity gate (a small CI script over registered catalogues). Tolerated until closed: no automated parity enforcement; `python-i18n` in legacy modules until [migration.md](migration.md) retires them.
