## 3. Assumptions Challenged

### Assumption 3.1: Rolling deployment with circuit breaker provides adequate deployment safety

- **Stated Norm:** "The rolling deployment model with circuit breaker provides adequate safety" (Alternatives §3)
- **Underlying Assumption:** A single ECS service with rolling deployment and circuit breaker is sufficient to prevent bad deployments from impacting users.
- **Challenge:** The circuit breaker only detects task launch failures and health check failures. It cannot detect application-level regressions (e.g., a bug that returns incorrect data but passes health checks). A canary or blue/green deployment would catch regressions through traffic comparison.
- **Evidence Strength:** ⭐⭐ Moderate
- **Counter-Evidence Found:** Yes — DORA research shows that canary/progressive delivery correlates with lower change fail rates in high-performing organizations. However, this applies at scale; for single-service deployments, the circuit breaker provides the primary safety net.
- **Confidence (ADR survives challenge):** 🟡 Moderate
- **Reviewer Notes:** The ADR explicitly acknowledges this limitation in Alternatives §3 and keeps the door open for canary/blue-green as a future evolution. For current operational scale (single ECS service, single team), the rolling deployment with circuit breaker is a defensible choice. The key mitigation is the pre-merge CI quality gates (S1) which catch regressions before they reach deployment.

### Assumption 3.2: Type check enforcement can be deferred until legacy refactoring completes

- **Stated Norm:** Type check classified as "Target" with enforcement deferred until legacy code is fully type-compliant
- **Underlying Assumption:** The ongoing refactoring effort will eventually bring the codebase to full mypy compliance, at which point the gate transitions to Mandatory.
- **Challenge:** Open-ended deferral risks the gate never becoming mandatory. Without a deadline or milestone, "deferred" becomes "permanent exception."
- **Evidence Strength:** ⭐⭐ Moderate
- **Counter-Evidence Found:** No — the refactoring effort is active and progressing. The `|| true` pattern is the correct interim approach (running the check for visibility without blocking).
- **Confidence (ADR survives challenge):** 🟡 Moderate
- **Reviewer Notes:** The ADR should ideally reference a milestone or Tier-5 migration ADR for the type compliance transition. However, tying it to the ongoing refactoring is acceptable as-is. The key insight is that the ADR *codifies the intent* to make this mandatory, which is better than the current implicit soft-fail.

### Assumption 3.3: This is the correct governance domain classification (infrastructure)

- **Stated Norm:** `governance_domain: infrastructure` — CI/CD pipelines are infrastructure-domain concerns.
- **Underlying Assumption:** The CI/CD pipeline is entirely within the infrastructure governance domain and does not require application-domain ADR authority.
- **Challenge:** CI quality gates (S1) enforce application-level standards (lint, format, type check, test). Doesn't the gate selection constitute an application-domain decision?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No — the distinction is clear. ADR-0062 (application domain) governs *what* is tested and *how* tests are structured. ADR-0081 (infrastructure domain) governs *whether the pipeline enforces these standards as blocking gates*. The pipeline is the enforcement mechanism, not the authority on test content. Analogy: a building code (application) defines fire safety requirements; the inspection process (infrastructure) enforces them.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** Clean separation. The `governance_domain: infrastructure` classification is correct.

### Assumption 3.4: IaC validation standards can be enforced in the same ADR as application deployment standards

- **Stated Norm:** S6 (IaC validation) is included alongside S1–S5 (application deployment pipeline standards).
- **Underlying Assumption:** Terraform pipeline validation and application deployment validation are related enough to be governed by the same Tier-2 Standard.
- **Challenge:** Terraform and application deployment are different pipeline concerns with different blast radii. Terraform changes affect infrastructure; application deployments affect the application. Should they be separate ADRs?
- **Evidence Strength:** ⭐⭐ Moderate
- **Counter-Evidence Found:** Partially — some organizations separate "CI/CD for infrastructure" from "CI/CD for applications" into distinct governance documents. However, both share the same fundamental principle: changes must be validated before production.
- **Confidence (ADR survives challenge):** 🟡 Moderate
- **Reviewer Notes:** The case for combining them is that they share the same pipeline infrastructure (GitHub Actions) and the same validation philosophy. Splitting into two ADRs would create unnecessary overhead for a team that manages both pipelines. If IaC complexity grows (e.g., multiple Terraform modules, Terragrunt environments), a future Tier-4 ADR could provide IaC-specific governance that inherits from S6. Acceptable as-is.

## 4. Failure Modes Identified

### Failure Mode 4.1: Circuit breaker passes but application is functionally broken

- **If Assumption 3.1 Fails:** A deployment passes ECS circuit breaker validation (tasks running, health checks green) but the application returns incorrect data or has a logic regression.
- **Platform Impact:**
  - Incident management workflow: Medium — incorrect bot responses to incidents
  - Access synchronization workflow: High — incorrect access decisions
  - Access request workflow: Medium — incorrect approval routing
  - Multi-provider integrations: Low — integration-level errors are typically caught by health checks
- **Probability Estimate:** Low (pre-merge CI gates catch most logic regressions; health check bypass requires a very specific failure mode)
- **Mitigation or Acceptance:** Accepted risk at current scale. Pre-merge CI gates (S1) are the primary defense. Future mitigation: smoke tests as part of the deployment pipeline (explicitly listed as a non-goal / deferred Tier-4 decision).

### Failure Mode 4.2: IaC standards delay critical infrastructure changes

- **If Assumption 3.4 Fails:** A critical Terraform change (e.g., security group fix during an incident) is blocked by fmt/validate gates.
- **Platform Impact:**
  - All workflows: High — incident response delayed
- **Probability Estimate:** Low (critical infrastructure changes during incidents are rare; most can be made via console or CLI as emergency override)
- **Mitigation or Acceptance:** Accepted risk. Emergency changes can bypass pipeline via `workflow_dispatch` or direct console/CLI actions. The ADR does not prohibit emergency overrides — it governs the standard pipeline path.

## 6. Scenario Validation Matrix

### Scenario 6.1: Developer pushes code change to main

- **Trigger:** PR merged to `main` branch
- **Expected Pipeline Behavior Under ADR-0081:**
  1. Pre-merge: lint ✅, format ✅, SAST (Bandit) ✅, type check ✅ (blocking), tests ✅ (blocking)
  2. Build: Docker image built, tagged with SHA+date
  3. Pre-push: container image scan (target — blocks on critical/high CVEs when implemented)
  4. Push: Image pushed to ECR
  5. Deploy: ECS force-new-deployment with circuit breaker enabled
  6. Validate: Circuit breaker monitors task launch + health checks; auto-rollback on failure
  7. Notify: Slack notification with SHA and outcome; Sentinel deployment log
- **Validation Result:** ✅ All standards exercised. Current gap: steps 3, 6 require implementation.

### Scenario 6.2: Deployment fails due to bad configuration

- **Trigger:** ECS task fails to start because required environment variable is missing
- **Expected Pipeline Behavior Under ADR-0081:**
  1. ECS circuit breaker Stage 1: Task fails to reach RUNNING state
  2. Failure counter incremented; after threshold reached, deployment marked FAILED
  3. Automatic rollback to last COMPLETED deployment
  4. Slack failure notification with deployment SHA and failure stage
  5. Pipeline exits with failure status (not `continue-on-error`)
- **Validation Result:** ✅ S3 (deployment validation) + S4 (failure notification) handle this scenario. Current gap: circuit breaker not enabled, `continue-on-error` still present.

### Scenario 6.3: Terraform change introduces infrastructure regression

- **Trigger:** PR modifies `terraform/ecs.tf` with incorrect task definition
- **Expected Pipeline Behavior Under ADR-0081:**
  1. Pre-merge: `terraform fmt -check` ✅, `terraform validate` ✅
  2. Plan output posted to PR for review
  3. Reviewer identifies the incorrect task definition in the plan diff
  4. PR blocked until plan is corrected
- **Validation Result:** ✅ S6 (IaC validation) catches the issue at plan review. Pre-merge validation already enforced via `cds-snc/terraform-plan`.

### Scenario 6.4: Two developers merge to main simultaneously

- **Trigger:** Two PRs merged within seconds, triggering two `build_and_deploy.yml` runs
- **Expected Pipeline Behavior Under ADR-0081:**
  1. GitHub Actions concurrency group serializes the deployments
  2. First deployment runs; second waits (or cancels first if `cancel-in-progress` is used)
  3. Only one deployment active at any time
- **Validation Result:** ✅ S5 (deployment concurrency) handles this scenario. Current gap: no concurrency group configured.
