## 3. Assumptions Challenged

### Assumption 3.1: blinker as registry-only is still valuable despite facade-owned dispatch

- **Stated Norm:** Standard 1 adopts blinker; Standard 4 prescribes facade-owned dispatch loop instead of `signal.send()`.
- **Underlying Assumption:** blinker provides sufficient value as a handler registry (weak refs, thread safety, signal deduplication, test utilities) to justify the dependency even though the facade owns dispatch.
- **Challenge:** If the facade owns the dispatch loop, blinker is just a `dict[str, list[Callable]]` with weak-ref support. Is a new dependency justified for registry-only value?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No — Standard 4.4 enumerates five specific capabilities blinker provides beyond a plain dict: (a) thread-safe receiver storage with weak references, (b) automatic garbage-collected handler cleanup, (c) `Namespace` for named signal deduplication, (d) `muted()`/`connected_to()` test utilities, (e) `receivers_for()` with liveness checks. Reimplementing these in custom code would be ~200 lines — the same code being replaced.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The "blinker as registry" concern was the Round 1 blocker's deeper implication. Standard 4.4 addresses it directly with a concrete enumeration. The facade dispatch loop is ~10 lines; the registry capabilities it replaces are ~200 lines. The tradeoff is sound.

### Assumption 3.2: The modules → packages migration creates genuine cross-cutting event needs

- **Stated Norm:** Context section states 8+ packages with cross-cutting concerns (audit, notifications, telemetry) justify an event system.
- **Underlying Assumption:** The migrated packages will actually need decoupled cross-cutting notifications, not just direct calls.
- **Challenge:** The SRE bot is an internal tool with a small team. Every consumer is known in advance. Could direct calls suffice?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** Partially — direct calls work when there are 1-2 consumers per event. But the codebase already demonstrates the duplication problem: `modules/groups/events/system.py` is a parallel event system created because infrastructure events didn't serve the module's needs. Without a governed event bus, each migrated package risks creating its own ad-hoc notification mechanism. The access package already has 15+ dispatch sites with declared cross-package contracts. Direct calls for 8 packages × 3 cross-cutting concerns would require 24+ injected dependencies in service constructors.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The "YAGNI / delete event system" position was thoroughly evaluated in Alternatives Considered (option 1). The rejection rationale is grounded in the specific migration ahead, not speculative extensibility.

### Assumption 3.3: Choreography saga is the right pattern for access request↔sync

- **Stated Norm:** Context cites choreography saga as the pattern for the access request→sync→request event loop.
- **Underlying Assumption:** Choreography (event-driven, no central coordinator) is preferable to orchestration (central coordinator) for this workflow.
- **Challenge:** The workflow has exactly two participants. An orchestrator would be simpler — one file that calls both services. Choreography adds event infrastructure complexity for a two-step process.
- **Evidence Strength:** ⭐⭐ Moderate
- **Counter-Evidence Found:** Yes — an orchestrator is simpler for two participants. However: (1) the orchestrator creates a coupling point — it must import both `request` and `sync` sub-packages, violating the package isolation that makes the access domain maintainable. (2) The choreography pattern is already declared in `common/events.py` with three cross-package event contracts. (3) If additional participants emerge (notifications, audit, telemetry), choreography scales without modifying the orchestrator.
- **Confidence (ADR survives challenge):** 🟡 Moderate — choreography is defensible but not the only viable pattern. This is a judgment call, not a technical requirement.
- **Reviewer Notes:** The ADR correctly positions this as a rationale for the event system, not as a binding standard. The choreography pattern is one use case; cross-cutting notifications are the stronger justification. If an orchestrator were preferred, the event system would still be needed for the 8 other packages' cross-cutting concerns.

### Assumption 3.4: Async migration justifies library choice over alternatives

- **Stated Norm:** Standard 5 prescribes async readiness. Alternatives analysis rejects PyPubSub due to lack of async support.
- **Underlying Assumption:** The sync → async migration will happen and requires async handler dispatch.
- **Challenge:** The async migration timeline is unspecified. If it's years away, choosing blinker for async support over PyPubSub (which has native error isolation) is optimizing for the wrong constraint.
- **Evidence Strength:** ⭐⭐ Moderate
- **Counter-Evidence Found:** Partially — the async migration is stated as a goal but has no governing ADR or timeline. However: (1) blinker beats PyPubSub on every criterion except native error isolation, and the facade closes that gap. (2) Choosing PyPubSub now and switching to blinker later for async would be two library migrations instead of one. (3) The facade insulates packages — if async never happens, blinker's sync dispatch works fine with zero waste.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** blinker is the better choice regardless of async timeline. PyPubSub's only advantage (native error isolation) is neutralized by the facade pattern. blinker's additional advantages (async support, larger community, Pallets ecosystem, test utilities) compound.

### Assumption 3.5: Current zero-consumer state is a pre-migration snapshot, not terminal

- **Stated Norm:** Codebase Audit states events are "structurally present but functionally hollow" and "will become load-bearing as packages are enabled."
- **Underlying Assumption:** The access package will be enabled in production and modules will migrate, creating real event consumers.
- **Challenge:** If the access package is never enabled, or modules never migrate, the event system remains dead infrastructure indefinitely.
- **Evidence Strength:** ⭐⭐ Moderate
- **Counter-Evidence Found:** No — the access package has extensive implementation (request service, sync service, adapters, DynamoDB store, comprehensive tests). The `modules/groups` deprecation is stated by the project owner. The migration map (ADR program) plans 8 module migrations. These are concrete plans, not speculative.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The Tier-5 retirement criteria (Standard 7) require at least one feature package to register handlers. If this never happens, the ADR remains in Draft/unretired state, which is an acceptable outcome — no production harm from an unused library dependency.

## 4. Failure Modes Identified

### Failure Mode 4.1: blinker maintenance discontinuation

- **If Assumption Fails:** Pallets Community Ecosystem stops maintaining blinker. No security patches, no Python version compatibility updates.
- **Platform Impact:**
  - Incident management workflow: None (blinker is infrastructure-internal)
  - Access synchronization workflow: Low (facade insulates; swap library)
  - Access request workflow: Low (same)
  - Multi-provider integrations: None (not in the dispatch path)
- **Probability Estimate:** Low — Pallets maintains Flask, Jinja, Werkzeug, Click, and MarkupSafe. blinker is part of this ecosystem. Last release Nov 2024, but that reflects maturity (blinker has been stable since 2010).
- **Mitigation:** The `EventDispatcher` facade isolates packages from blinker. Library substitution requires changing only facade internals (~50 lines). No feature code changes needed.

### Failure Mode 4.2: Async migration never happens — Standard 5 is unnecessary

- **If Assumption Fails:** The codebase remains sync-only indefinitely. `send_async()` and mixed handler detection are never used.
- **Platform Impact:**
  - All workflows: None — sync dispatch works fine. Standard 5 is additive (describes future evolution), not a current implementation requirement.
- **Probability Estimate:** Medium — async migration depends on team capacity and priorities.
- **Mitigation:** Standard 5 is forward-compatible. blinker's sync dispatch (`receivers_for()` + call) works identically whether async handlers exist or not. No wasted effort — the standard documents evolution path but requires no implementation until triggered.

### Failure Mode 4.3: Facade dispatch loop introduces bugs not present in blinker's tested `send()`

- **If Assumption Fails:** The custom dispatch loop in the facade has edge-case bugs (weak-ref races, signal mutation during iteration, re-entrancy) that blinker's tested `send()` handles correctly.
- **Platform Impact:**
  - All workflows using events: Medium — handlers might be skipped or called incorrectly.
- **Probability Estimate:** Low — the dispatch loop is ~10 lines iterating `receivers_for()` (which handles weak-ref liveness and cleanup). The loop is simpler than blinker's `send()` because it doesn't collect return values.
- **Mitigation:** (1) `receivers_for()` is a blinker-provided generator that handles all weak-ref complexity. (2) The facade dispatch loop is unit-tested per Implementation Guidance. (3) The existing custom dispatcher already implements this exact pattern (try/except per handler in a for loop) — it's proven code.

## 7. Tradeoffs Accepted

### Tradeoff 7.1: Facade-owned dispatch vs. library-native dispatch

- **Chosen:** Facade iterates `receivers_for()` with per-handler try/except (Standard 4).
- **Rejected:** Direct `signal.send()` / `signal.send_async()` dispatch.
- **Rationale:** blinker propagates first exception, skipping remaining handlers. Error isolation is a hard requirement for fire-and-forget side-effects. Django solves this with `Signal.send_robust()` — blinker chose not to implement this variant.
- **Risk Accepted:** Facade dispatch loop may have edge cases not covered by blinker's tested `send()`. Mitigated by: `receivers_for()` handles weak-ref complexity; loop is ~10 lines; unit-tested.
- **Contingency:** If bugs surface, the facade can switch to a patched blinker (adding `send_robust()` upstream) or to PyPubSub (native error isolation) with no feature code changes.

### Tradeoff 7.2: New dependency vs. continued custom code

- **Chosen:** Add `blinker` dependency.
- **Rejected:** Continue maintaining custom `EventHandlerRegistry` + dispatch loop.
- **Rationale:** ADR-0045 P7 mandates highest applicable delegation tier. blinker replaces ~200 lines of custom registry code. The library is stable (since 2010), well-maintained (Pallets), and minimal (no transitive deps).
- **Risk Accepted:** External dependency maintenance risk. Mitigated by: Pallets ecosystem backing; facade enables library substitution.
- **Contingency:** If blinker is abandoned, facade swap to PyPubSub or custom registry (revert to Tier 3 with ADR-0045 P7 exception documented).

### Tradeoff 7.3: Building event infrastructure before consumers exist

- **Chosen:** Establish event infrastructure now (during greenfield `packages/` development).
- **Rejected:** Defer until first concrete consumer emerges (YAGNI).
- **Rationale:** Building now costs less than retrofitting later. The access package already has 15+ dispatch sites and 3 cross-package event contracts. The modules → packages migration will create 8+ packages with cross-cutting concerns. Establishing the pattern during greenfield development avoids the "each package invents its own notification mechanism" duplication already demonstrated by `modules/groups/events/system.py`.
- **Risk Accepted:** If migration stalls, event infrastructure sits unused. Mitigated by: blinker is zero-config; unused dependency has negligible runtime cost.
- **Contingency:** Tier-5 retirement criteria require at least one consumer. If never met, ADR remains unretired — no production harm.
