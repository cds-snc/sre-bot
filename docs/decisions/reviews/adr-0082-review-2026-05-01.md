## 3. Assumptions Challenged

### Assumption 3.1: Q Developer natively formats CloudWatch alarms without custom code

- **Stated Norm:** "Q Developer natively solves the core problems (raw JSON → formatted messages, managed delivery) without custom Lambda code."
- **Underlying Assumption:** CloudWatch alarm SNS payloads are recognized and formatted by Q Developer automatically.
- **Challenge:** Could Q Developer treat CloudWatch alarm SNS messages as unformatted generic JSON (same as the current webhook problem)?
- **Evidence Strength:** ⭐ Strong — AWS documentation explicitly lists CloudWatch Alarms as a supported service with native formatting. The EventBridge note confirms this: "Event notifications from CloudWatch Alarms are NOT supported via EventBridge rules" — SNS-direct is the specified path, confirming CloudWatch Alarms receive first-class native support, not generic passthrough.
- **Counter-Evidence Found:** No.
- **Confidence:** 🟢 High
- **Notes:** The ADR correctly identifies this. CloudWatch Alarms are a Tier-1 natively supported service in Q Developer.

---

### Assumption 3.2: Terraform `aws_chatbot_slack_channel_configuration` can fully provision the integration

- **Stated Norm:** "Terraform-native. Full resource support via `aws_chatbot_slack_channel_configuration`. Infrastructure-as-code compliant per ADR-0081 S6."
- **Underlying Assumption:** All required configuration can be expressed as Terraform code; no manual steps beyond the one-time workspace authorization.
- **Challenge:** Are there additional manual setup steps not captured in the ADR (Slack app approval, IAM service-linked role, etc.)?
- **Evidence Strength:** ⭐ Strong — Terraform docs confirm the resource fully manages the channel configuration post-workspace-authorization. The `aws_chatbot_slack_workspace` data source enables referencing the authorized workspace ID in code.
- **Counter-Evidence Found:** Only the one-time workspace authorization is manual — confirmed and already noted in the ADR. All subsequent configuration (channel mapping, SNS topic ARNs, IAM role, logging, guardrails) is Terraform-managed.
- **Confidence:** 🟢 High
- **Notes:** The ADR accurately represents the automation boundary.

---

### Assumption 3.3: Q Developer's fixed message format is sufficient for the fallback use case

- **Stated Norm:** "Q Developer's fixed format — alarm name, state, metric, threshold, region — provides sufficient context for an operator to begin investigation."
- **Underlying Assumption:** An alarm notification containing alarm name + state + metric + threshold + region is operationally actionable when the bot is down.
- **Challenge:** If the alarm fires at 3 AM and the description is generic ("SRE Bot Errors alarm"), would the operator know what to do?
- **Evidence Strength:** ⭐⭐ Moderate — the ADR mitigates this by requiring alarm descriptions to encode actionable context. The sufficiency depends entirely on the quality of `alarm_description` values set in Terraform.
- **Counter-Evidence Found:** No counter-evidence that Q Developer's format is insufficient for a "bot is down, begin investigation" signal. The concern is execution quality, not architectural validity.
- **Confidence:** 🟢 High (architectural assumption holds; execution risk is mitigated and noted)
- **Notes:** The ADR correctly acknowledges this tradeoff and mitigation. No revision required.

---

### Assumption 3.4: The SLACK_BOT_TOKEN constraint applies to the chosen approach

- **Stated Norm (in Constraints):** "Slack's bot token (`SLACK_BOT_TOKEN`) is available as the app identity for posting messages (user-confirmed: same bot token as SRE Bot app)."
- **Underlying Assumption:** The chosen approach (Option A, Q Developer) uses the SRE Bot's bot token to post to Slack.
- **Challenge:** Q Developer uses its own Slack app OAuth (AWS Chatbot). It does not use `SLACK_BOT_TOKEN`. The token constraint is applicable exclusively to Option B (Lambda + `chat.postMessage`). For the chosen approach, the Slack identity will be the AWS Chatbot application, not the SRE Bot.
- **Evidence Strength:** ⭐ Strong — Terraform docs show `slack_team_id` and `slack_channel_id` are the identity parameters; no token field exists. Q Developer uses the AWS Chatbot Slack app OAuth, which is authorized during the one-time workspace setup.
- **Counter-Evidence Found:** Yes — the constraint as written is factually incorrect for Option A.
- **Confidence:** 🔴 Low (assumption fails for the chosen approach)
- **Reviewer Notes:** This is a precision issue, not an architectural blocker. The consequence is that Slack notifications from Q Developer will appear as the "AWS" Slack app, not the SRE Bot. This is architecturally desirable (independent identity = independent mechanism), but the Constraints section implies the SRE Bot's identity will be used. The constraint should be rewritten to (a) apply the `SLACK_BOT_TOKEN` note to Option B only, and (b) clarify that for Option A, Slack messages appear from the AWS Chatbot app.

---

### Assumption 3.5: Least-privilege IAM is fully addressed by the IAM role policy

- **Stated Norm:** "A dedicated IAM role for Q Developer with least-privilege permissions (CloudWatch read-only, no administrative access). Guardrail policies applied via `guardrail_policy_arns`."
- **Underlying Assumption:** The `guardrail_policy_arns` requirement as stated in the ADR is sufficient to prevent privilege escalation from Slack.
- **Challenge:** The Terraform resource's `guardrail_policy_arns` **defaults to `AdministratorAccess` if not set**. "Not set" is a valid Terraform state (the argument is optional). Anyone in the connected Slack channel could then run AWS CLI commands with admin privileges directly from Slack. The ADR does not mandate that this attribute must be set to a restrictive policy — it only notes that guardrails are "applied" via the attribute.
- **Evidence Strength:** ⭐ Strong — Terraform documentation explicitly states: "The AWS managed `AdministratorAccess` policy is applied by default if this is not set."
- **Counter-Evidence Found:** Yes — the ADR does not require a specific restrictive guardrail policy. An implementation following the ADR literally could omit `guardrail_policy_arns`, which silently grants AdministratorAccess to all Slack channel members.
- **Confidence:** 🔴 Low (assumption fails — the guardrail requirement is insufficiently specific)
- **Reviewer Notes:** **BLOCKER.** This is a security-critical omission. The ADR must explicitly require `guardrail_policy_arns` to be set to a deny-all or read-only policy (e.g., `arn:aws:iam::aws:policy/ReadOnlyAccess`). For an alerting-only use case, no interactive AWS CLI commands from Slack are needed. The policy must deny all `chatbot:*` interactive operations. Leaving this to implementation discretion violates ADR-0064 (Security and Rate-Limiting API Protection) and OWASP Principle of Least Privilege.

## 4. Failure Modes Identified

### Failure Mode 4.1: `guardrail_policy_arns` not set — Slack channel members can run AWS CLI commands with admin access

- **If Assumption Fails:** An engineer implements the Q Developer Terraform resource without setting `guardrail_policy_arns`. AWS applies AdministratorAccess by default. Any Slack workspace member in the ops channel can execute arbitrary AWS CLI commands from Slack (e.g., delete ECS services, modify IAM roles, read Secrets Manager values).
- **Platform Impact:**
  - Security posture: Critical
  - Incident blast radius: High (any Slack user in the channel gains AWS admin access)
  - ADR-0064 compliance: Violated
- **Probability Estimate:** High — the argument is optional; without an explicit ADR mandate, it will likely be omitted in implementation.
- **Mitigation:** Add an explicit implementation requirement: `guardrail_policy_arns` MUST be set to a deny-all interactive policy. Suggested: create a dedicated managed policy that denies all actions except those required for passive alarm receipt.

---

### Failure Mode 4.2: Workspace authorization not completed before `terraform apply`

- **If Assumption Fails:** The `aws_chatbot_slack_workspace` data source lookup fails because the workspace has not been manually authorized. `terraform plan` will succeed (the data source may not error at plan time); `terraform apply` will fail.
- **Platform Impact:** Deployment blocked until manual step is completed. Not a runtime failure — an operational sequencing issue.
- **Probability Estimate:** Low — the manual prerequisite is documented in the ADR. First-deploy risk only.
- **Mitigation:** The ADR already documents this as a one-time action. Mitigated by clear operational runbook at implementation time.

---

### Failure Mode 4.3: Q Developer service outage during bot failure

- **If Assumption Fails:** Amazon Q Developer is unavailable at the same time the SRE Bot fails. No fallback path exists.
- **Platform Impact:** The fallback path itself fails. Ops team is not notified.
- **Probability Estimate:** Low — simultaneous failure of ECS (hosting the bot) and Q Developer (AWS managed service) is unlikely.
- **Mitigation:** ADR already acknowledges this risk and reserves Option B as an escalation. Acceptable risk for Phase 1.

## 6. Revision Requirements

### R1.1 — BLOCKER: Explicitly mandate restrictive `guardrail_policy_arns`

**Location:** Decision → Implementation requirements → item 5 (IAM role)  
**Required change:** Add an explicit statement that `guardrail_policy_arns` MUST be set to a restrictive policy that denies all interactive AWS CLI operations from Slack. The default (`AdministratorAccess`) must never be used for an alerting-only channel. Suggested minimum: `arn:aws:iam::aws:policy/ReadOnlyAccess` or a custom deny-all policy. This should be elevated from a parenthetical note to a mandatory implementation requirement.

**Why it's a blocker:** Leaving `guardrail_policy_arns` unset silently grants AdministratorAccess to every member of the connected Slack channel. This violates OWASP Least Privilege and ADR-0064. It is a zero-effort omission with a critical blast radius.

---

### R1.2 — PRECISION: Correct SLACK_BOT_TOKEN constraint scope

**Location:** Context → Constraints (third bullet)  
**Required change:** The constraint "Slack's bot token (`SLACK_BOT_TOKEN`) is available as the app identity for posting messages (user-confirmed: same bot token as SRE Bot app)" is only applicable to Option B (Lambda + `chat.postMessage`). For Option A (Q Developer), the Slack identity is the AWS Chatbot Slack app — a separate Slack application managed by AWS. The constraint should be moved to the Option B description and a clarifying note added that Q Developer notifications appear from the AWS Chatbot app, not the SRE Bot.

**Why it matters:** If left as a general constraint, it implies the SRE Bot's identity is used for the chosen approach, which is false. It could also mislead an implementer into attempting to configure a bot token in the Q Developer Terraform resource (which doesn't support one).

---

### R1.3 — MINOR: Clean up Option A assessment P-code references

**Location:** Decision → Alternatives → Option A Assessment row  
**Required change:** The assessment row references "P1", "P3", "P2", "P4", "P5", "P6" codes ("Solves P1 (formatted messages), P6 (governance). Partially addresses P3...") which no longer correspond to any labeled problems in the current Context section. These should be updated to plain-language references or removed.
