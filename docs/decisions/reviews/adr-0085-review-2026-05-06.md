## 3. Assumptions Challenged

### Assumption 3.1: "The `__init__.py` barrel is a Service Locator"

- **Stated Norm:** "The problem is that the three-file ceremony (ADR-0056 Standard 4) caused `__init__.py` to act as a **Service Locator**: any module can obtain any service by importing from a single address without declaring a dependency on the owning package."
- **Underlying Assumption:** That a Python `__init__.py` re-export module is functionally equivalent to Seemann's Service Locator pattern and therefore inherits all its architectural harms.
- **Challenge:** Seemann's Service Locator is a *runtime registration pattern* — a global dictionary that code queries dynamically (`locator.get(IEmailService)`). A Python `__init__.py` re-export is a *static compile-time namespace alias*. The coupling risk in a barrel is real (ownership obscured, import cost forced), but the antipattern is more precisely "aggregation of non-owned symbols" than "Service Locator." The Composition Root / Service Locator distinction as framed in the table is accurate for the *architectural intent* but imprecise for the *technical mechanism*. Over-stating the analogy could lead to unnecessary scope expansion (e.g., arguing that any multi-symbol `__init__.py` is a Service Locator).
- **Evidence Strength:** ⭐ Strong — the practical concern is well-evidenced; the Seemann citation is slightly imprecise but does not affect the decision.
- **Counter-Evidence Found:** No — the harmful effects of the current barrel (import cost, ownership obscurity) are observable in the code. The characterization issue is terminological, not substantive.
- **Confidence (ADR survives challenge):** 🟢 High — the framing imprecision does not materially affect the decision. The ownership violation and import cost are the primary justifications and are independently sound.
- **Reviewer Notes:** Recommend a one-sentence qualification: "The barrel exhibits the *effect* of a Service Locator — any consumer can obtain any service from a single address — without the runtime-registration mechanism Seemann describes." This would pre-empt future terminological disputes.

---

### Assumption 3.2: "`from infrastructure.services import hookimpl` triggers loading of boto3, google-auth, and MaxMind SDKs"

- **Stated Norm:** "Every feature package `__init__.py` contains `from infrastructure.services import hookimpl`. This single import executes the full `__init__.py`, which imports `providers.py`, which imports every infrastructure service module — including heavy SDK wrappers (boto3, google-auth, MaxMind)."
- **Underlying Assumption:** That `providers.py` has module-level imports of heavy SDK client classes, causing those SDKs to be loaded when `providers.py` is imported.
- **Challenge:** This was validated against the actual `providers.py` file. Lines 13–60 of `providers.py` include module-level imports of `AWSClients`, `GoogleWorkspaceClients`, `MaxMindClient`, `SlackClientFacade`, `TeamsClientFacade`, and `DiscordClientFacade`. These class imports execute when `providers.py` is first loaded, which in turn runs those modules' module-level code (e.g., `from geoip2 import database` in maxmind, `import boto3` in AWS clients). The claim is **verified by code inspection**.

  **Partial nuance:** S6.2 states "Provider functions in `providers.py` that construct heavy clients must use lazy initialization inside the `@lru_cache` function body, not at module import time." This correctly prevents *instance construction* from happening at import time, but it does **not** prevent *class definition imports* (the `from infrastructure.clients.maxmind import MaxMindClient` line) from executing. After barrel dissolution, a consumer doing `from infrastructure.services.providers import get_maxmind_client` will still load all of `providers.py` and all its module-level imports, including every SDK class. The performance win is scoped to the specific case where `hookimpl` is the only needed symbol — once any consumer imports from `providers.py`, the full module-level cost is paid.

- **Evidence Strength:** ⭐ Strong — validated by code inspection.
- **Counter-Evidence Found:** No — the claim is accurate. The nuance is that it only applies to the specific `hookimpl` import path, not to any consumer that actually needs a provider.
- **Confidence (ADR survives challenge):** 🟡 Moderate — the performance claim in S6.1 is valid. S6.2's "lazy initialization" requirement is technically correct but incomplete: it governs constructor calls, not module-level class imports. If `providers.py` itself has SDK-loading module-level imports (it does), then `from infrastructure.services.providers import get_event_dispatcher` still pays heavy costs. The ADR should acknowledge this distinction.
- **Reviewer Notes:** S6.2 should be clarified to explicitly distinguish between (a) lazy *instance construction* in provider function bodies (already enforced by `@lru_cache`) and (b) *deferred module imports* (importing the client class inside the function body rather than at module scope). The latter is not currently required by the ADR and is not done in `providers.py`. This gap means the import-time benefit is only realized for the specific `hookimpl` path, not for any code that needs infrastructure providers.

### Assumption 3.3: "Dissolving `dependencies.py` removes the Service Locator symptom"

- **Stated Norm (S3, Rationale):** "Centralized alias declarations create the Service Locator symptom. Every import from `dependencies.py` triggers the full barrel load. Consumer-owned aliases import only the specific provider function needed."
- **Underlying Assumption:** That consumer-owned `Annotated[Protocol, Depends(get_x)]` declarations are architecturally superior to centralized aliases, and that the migration from centralized to consumer-owned DI aliases is the correct fix for the import cost problem.
- **Challenge:** ADR-0056 Standard 4 states *explicitly* and *recently* (accepted 2026-04-29, last updated 2026-05-04): "The three-file pattern for adding a new infrastructure service remains the canonical approach." Ceremony rules C2 and C3 mandate that all aliases and providers be re-exported through `__init__.py`. ADR-0056 was *updated on 2026-05-04* — two days before ADR-0085 was authored — and its ceremony was intentionally retained. ADR-0085 Standard 3 dissolves `dependencies.py` and forbids new aliases (S3.1), directly contradicting these retained provisions. **ADR-0085 does not supersede ADR-0056, does not list ADR-0056 in its `supersedes` metadata field, and does not acknowledge this contradiction in its text.**

  Additionally, ADR-0056 Standard 3.2 states self-contained providers "must still… Be re-exported through `infrastructure.services.__init__.py` so the single consumption surface (ADR-0048 B2) is maintained." ADR-0085 Standard 1 forbids re-exporting non-owned symbols. For self-contained providers (defined outside `infrastructure/services/` but re-exported for B2 compliance), these two standards are irreconcilable.

- **Evidence Strength:** ⭐ Strong — both ADRs are in-scope, recently authored, and contain explicit contradictory text.
- **Counter-Evidence Found:** Yes — ADR-0056 S3.2 and S4 C2/C3 directly contradict ADR-0085 S1 and S3. This is the primary blocking finding.
- **Confidence (ADR survives challenge):** 🔴 Low — unless the contradiction is resolved by explicit supersession of ADR-0056 S3.2 and S4 C2/C3, or by amendment of ADR-0056 to align with the new barrel governance.
- **Reviewer Notes:** The resolution requires one of two paths: (a) Add ADR-0056 to ADR-0085's `supersedes` field with a clear statement of which provisions are superseded (specifically S3.2 re-export requirement and S4 C2/C3 ceremony rules for `__init__.py`); or (b) Amend ADR-0056 to remove the conflicting provisions and add a reference to ADR-0085 for barrel governance. Path (a) is simpler and more traceable. This is the **primary blocking finding** for the REVISE gate outcome.

### Assumption 3.4: "Deep imports from `providers.py` are consistent with ADR-0048 Boundary 2"

- **Stated Norm (S2):** "Consumers must import provider functions, Protocol types, and service facades from their defining modules, not from the barrel. CORRECT: `from infrastructure.services.providers import get_event_dispatcher`"
- **Underlying Assumption:** That importing directly from `infrastructure.services.providers` satisfies ADR-0048 Boundary 2's "single, defined injection surface" requirement.
- **Challenge:** ADR-0048 Boundary 2 states: "Application code imports only from this boundary — never from infrastructure internals, client packages, or configuration modules directly." `infrastructure.services.providers` is an internal module of the `infrastructure.services` package, not a public API boundary. The current boundary is `infrastructure.services` (the package namespace). After dissolution, `providers.py` becomes the de facto injection surface for non-HTTP consumers, and `infrastructure.security.current_user` becomes the surface for auth consumers. The ADR implicitly redefines the injection surface without explicitly acknowledging this redefinition of B2's "boundary."

  The practical risk is that "deep imports from any infrastructure module" could be confused with the permitted deep imports specified in S2. A future developer might reasonably ask: if `from infrastructure.services.providers import get_x` is permitted, why not `from infrastructure.storage.service import DynamoDBStorageService`? The ADR needs to clearly define what constitutes the permitted injection surface after dissolution.

- **Evidence Strength:** ⭐⭐ Moderate — ADR-0048 B2 language is ambiguous about whether the "boundary" is the package namespace or any explicitly governed path.
- **Counter-Evidence Found:** Yes — ADR-0048 B2 states "never from infrastructure internals." `providers.py` is an internal module of `infrastructure.services`. The ADR's S2 mandates importing from it without acknowledging this potential B2 conflict.
- **Confidence (ADR survives challenge):** 🟡 Moderate — the ADR can survive this challenge if it adds an explicit statement that the permitted injection surface after dissolution includes `infrastructure.services.providers`, `infrastructure.services.dependencies` (until dissolved), and `infrastructure.<domain>.protocol` modules. This would constitute a precise redefinition of B2's boundary for the post-dissolution state.
- **Reviewer Notes:** Add an explicit "Injection Surface Definition" statement to the Compliance and Boundaries section, listing the permitted deep import paths and clarifying that `infrastructure.<domain>.service`, `infrastructure.<domain>.dynamodb`, etc. (concrete implementations) remain prohibited for feature consumers. This resolves the B2 ambiguity.

### Assumption 3.5: "The retained symbols list is complete"

- **Stated Norm (S1):** "Symbols retained in `__init__.py` (owned by this package): `hookimpl`, `get_plugin_manager`, `discover_and_init_features`, `collect_feature_i18n_resources`"
- **Underlying Assumption:** That these four symbols from `infrastructure.services.plugins` are the complete set of symbols owned by `infrastructure/services/` and legitimately retained in `__init__.py`.
- **Challenge:** The current `infrastructure/services/__init__.py` re-exports `register_feature_integrations` from `infrastructure.services.plugins` (visible in the `__all__` list). This symbol comes from `infrastructure.services.plugins` — the same source module as `hookimpl` — and therefore qualifies as owned by the `infrastructure/services/` package. It is absent from the retained symbols list in S1. If it is intentionally excluded (e.g., it will be deprecated or moved), the rationale should be stated. If it was accidentally omitted, the list is incomplete.
- **Evidence Strength:** ⭐ Strong — validated by code inspection of the current `__init__.py`.
- **Counter-Evidence Found:** Yes — `register_feature_integrations` is present in the current `__init__.py` and sourced from `infrastructure.services.plugins`. It is not in the ADR's retained list.
- **Confidence (ADR survives challenge):** 🟡 Moderate — this is a minor completeness issue. The fix is straightforward: either add `register_feature_integrations` to the retained list or add a note stating it is intentionally excluded with rationale.
- **Reviewer Notes:** Audit the full current `__init__.py` against the retained symbols list before finalizing. Also confirm whether `t` (the translation shorthand re-exported from `providers.py`) is intentionally excluded — if it is widely used by feature packages via `from infrastructure.services import t`, a migration path for this specific symbol should be mentioned.

## 4. Failure Modes Identified

### Failure Mode 4.1: ADR-0056 Ceremony Conflict → Contradictory Implementation Guidance

- **If Assumption Fails:** If the ADR-0056 / ADR-0085 contradiction is not resolved before implementation begins, developers will receive conflicting instructions. ADR-0056 S4 C2/C3 tells developers to re-export everything through `__init__.py`; ADR-0085 S1 prohibits this. Code review will produce contradictory feedback depending on which ADR the reviewer consults. New infrastructure services added during migration will be wired inconsistently.
- **Platform Impact:**
  - Incident management workflow: Low — runtime behavior is not affected; only import paths change.
  - Access synchronization workflow: Medium — access sync providers currently import from `infrastructure.services` (barrel); contradictory guidance during migration could result in partially migrated imports that break under the new barrel regime before migration completes.
  - Access request workflow: Medium — same as access sync.
  - Multi-provider integrations (Slack, Teams, GWS, AWS, GitHub): Low — import path changes are mechanical; no behavioral impact.
- **Probability Estimate:** High — without an explicit supersession, both ADRs will be consulted and they will give contradictory guidance.
- **Mitigation or Acceptance:** Resolve by adding explicit supersession of ADR-0056 S3.2 and S4 C2/C3 to ADR-0085. This converts a governance conflict into a documented, intentional architectural evolution.

---

### Failure Mode 4.2: Incomplete Import-Time Performance Fix → Developer Expectation Mismatch

- **If Assumption Fails:** If developers implement S6 believing that barrel dissolution eliminates all heavy SDK loading at import time, they may be surprised to find that `from infrastructure.services.providers import get_maxmind_client` still loads all of `providers.py`'s module-level imports (including `MaxMindClient`, `AWSClients`, etc.). Tests may still experience slow startup when a single provider import is needed.
- **Platform Impact:**
  - Incident management workflow: Low — production runtime is unaffected.
  - Access synchronization workflow: Low — full provider import is expected in sync context.
  - Access request workflow: Low — same.
  - Multi-provider integrations: Low — full provider import is expected.
  - Test suite: Medium — test files that import a single provider for isolated unit tests will still pay the full `providers.py` module load cost, limiting the test isolation benefit.
- **Probability Estimate:** Medium — likely to affect test authors who expect deep imports to reduce load scope.
- **Mitigation or Acceptance:** Clarify S6.2 to distinguish (a) lazy constructor calls inside `@lru_cache` functions (already enforced) from (b) deferred module imports for heavy client classes inside provider function bodies (not yet required). The full S6 benefit is only achieved if `providers.py` is refactored to use `import` statements inside provider function bodies for heavy SDKs. This is a follow-up scope item and should be documented explicitly.

---

### Failure Mode 4.3: S5.3 Unenforceability → Compatibility Re-exports Used by Non-Frozen-Zone Code

- **If Assumption Fails:** S5.3 states "No non-frozen-zone code may import these compatibility re-exports." If no lint rule or CI check enforces this, non-frozen-zone code may progressively adopt the compatibility re-exports as a convenience shortcut, extending the lifetime of the barrel beyond the frozen zone's retirement.
- **Platform Impact:**
  - All workflows: Low at first, Medium over time — the compatibility re-export period could become permanent if usage accumulates without enforcement.
- **Probability Estimate:** Medium — absent enforcement, convenience imports tend to proliferate.
- **Mitigation or Acceptance:** The ADR's Enforcement section proposes a lint rule for `infrastructure.services` imports referencing non-owned symbols. This lint rule should explicitly cover compatibility re-exports from S5.1 to ensure non-frozen-zone code does not adopt them. The lint rule implementation should be tracked as a blocking implementation deliverable.

## 7. Tradeoffs Accepted

### Tradeoff 7.1: Import Explicitness vs. Consumption Ergonomics

- **Chosen:** Deep imports from defining modules (`from infrastructure.services.providers import get_x`), trading ergonomic brevity for explicit ownership.
- **Rejected:** Barrel retention with lazy imports or partial dissolution.
- **Rationale:** The barrel's import cost affects the plugin registration path (critical), and the ownership violation creates governance friction. Explicit imports are recommended by PEP 8 and Google Style Guide. The verbosity cost is real but bounded.
- **Risk Accepted:** Developers must know (or look up) the defining module for each symbol. Deep import paths couple consumers to the internal file structure of `infrastructure/services/`. If `providers.py` is refactored (e.g., split into per-service provider modules), all deep import consumers break.
- **Contingency:** S4 placement criteria include "change isolation — moving must not increase the blast radius of provider changes on consumers." Any future `providers.py` restructuring requires a migration plan per S4.4. The blast-radius risk is accepted but governed.

---

### Tradeoff 7.2: Frozen Zone Backward Compatibility vs. Clean Dissolution

- **Chosen:** Retain compatibility re-exports in `__init__.py` for the frozen zone (`app/modules/`), marked with `# COMPAT` markers.
- **Rejected:** Immediate dissolution including frozen zone breakage.
- **Rationale:** The frozen zone is a zero-touch policy area; breaking it to achieve architectural purity is disproportionate to the benefit. Compatibility re-exports with markers provide a clear deprecation signal.
- **Risk Accepted:** The compatibility period may extend indefinitely if the frozen zone thaw is deprioritized. The `# COMPAT` markers have no enforcement mechanism to prevent adoption by non-frozen-zone code (S5.3 risk, per Failure Mode 4.3).
- **Contingency:** S5.3 enforcement via lint rule. The lint rule scope must explicitly cover compatibility re-exports (this is not currently stated in the Enforcement section).

---

### Tradeoff 7.3: `dependencies.py` Dissolution vs. DI Alias Traceability

- **Chosen:** Dissolve `dependencies.py`; consumer-owned DI aliases.
- **Rejected:** Retain `dependencies.py` ceremony (per ADR-0056 S4).
- **Rationale:** Centralized alias declarations force every HTTP handler to import from `dependencies.py`, which triggers the full barrel load. Consumer-owned aliases import only the specific provider function needed.
- **Risk Accepted:** Loss of the single-location alias inventory. When reviewing DI usage across the codebase, developers must search for `Annotated[Protocol, Depends(get_x)]` patterns across consumer files rather than consulting one file. This increases discoverability cost.
- **Contingency:** IDE "go to definition" and search tooling mitigate the discoverability cost for individual symbols. **Note:** This tradeoff is currently irreconcilable with ADR-0056 S4 without explicit supersession — see Follow-Up Actions.

## 9. Binary Gate Outcome

**GATE DECISION:**

🔴 **REVISE** → ADR-0085 requires authoring revision; return to Step 5-9 author team with feedback

**Primary Blockers:**

1. **ADR-0056 S3.2 / S4 contradiction (unresolved, High severity):** ADR-0085 Standard 1 ("barrel re-exports owned symbols only") and Standard 3 ("dissolve `dependencies.py`") directly contradict ADR-0056 Standard 3.2 ("self-contained providers re-exported via `__init__.py`") and Standard 4 ceremony C2/C3 ("three-file pattern retained; all aliases and providers in `__init__.py`"). ADR-0085 carries no supersession declaration. Both records are simultaneously in-force with contradictory mandatory standards.

2. **`constrained_by` / `impacts` metadata conflict for ADR-0056 (unresolved, Medium severity):** ADR-0085 lists ADR-0056 in `constrained_by` (must not violate) and `impacts` (changes it). This is logically inconsistent. The metadata must be corrected to reflect the actual governance relationship — supersession or amendment — before the ADR can be formally accepted.

3. **ADR-0048 B2 injection surface redefinition unacknowledged (unresolved, Medium severity):** Deep imports from `infrastructure.services.providers` may conflict with ADR-0048 B2's "never from infrastructure internals" boundary. The ADR must either explicitly redefine the B2 boundary for the post-dissolution state or add a cross-reference to ADR-0048's amendment. Without this, B2 compliance becomes ambiguous for every feature package that adopts the new deep import pattern.
