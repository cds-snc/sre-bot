## 3. Assumptions Challenged

### Assumption 3.1: DynamoDB Entity Status Is Sufficient for Multi-Step Continuation Without an In-Process Coordinator

- **Stated Norm:** "The domain entity's `status` field in DynamoDB IS the multi-step continuation model. There is no separate `WorkflowState` entity, coordinator class, or in-process state machine." (Standard 8 S3)
- **Challenge:** For interactions with many possible states and complex transition conditions (e.g., a multi-approver flow with time-boxed windows), relying solely on DynamoDB `status` may cause logic duplication across multiple handler functions (each must re-validate the current status before transitioning). A coordinator reduces this repetition.
- **Evidence Strength:** ⭐ Low — the challenge applies to complex multi-step flows. However, this pattern is well-established in event-driven systems and is the documented approach for DynamoDB-backed workflows (AWS Builders Library, Richardson's Saga).
- **Counter-Evidence:** Standard 8 S4 explicitly governs this: "Domain status transitions are pure functions in `service.py`. A legal transition table is defined per entity type." The transition table in `service.py` is the coordinator — it is just expressed as data (a transition table) rather than a stateful object. This is a more testable and more 12-Factor-compliant pattern. Source: `tmp/target-state-architecture-stateless-2026-05-06.md`.
- **Confidence (ADR survives):** 🟢 High — the pure-function transition table is a correct and well-understood pattern.

---

### Assumption 3.2: Per-Platform Hookspecs Are the Correct Registration Model at Scale

- **Stated Norm:** "Registration uses per-platform hooks with concrete platform service types to preserve platform-native type signatures." (Standard 3 K1) Four hookspecs: `register_slack_interactions`, `register_teams_interactions`, `register_routes`, `register_event_handlers`.
- **Challenge:** With 4 platforms at current and a potential 5th (Discord), a new platform requires adding a new hookspec. This is an extension point that requires changes to `infrastructure/hookspecs/features.py` for each new platform.
- **Evidence Strength:** ⭐⭐ Moderate — hookspec proliferation is a real scalability concern at 6+ platforms.
- **Counter-Evidence:** Pluggy's own design encourages per-hookspec granularity. ADR-0078 (Platform Services Architecture) explicitly rejected the InteractionProvider Protocol precisely because platform APIs are asymmetric — no shared Protocol is appropriate. The per-platform hookspec model is the correct consequence of that decision. Research `tmp/research-plugin-registration-granularity-2026-05-06.md` confirms: one plugin per module (umbrella `__init__.py`), not per sub-capability. The hookspec count (4) is within normal Pluggy usage. Discord is a stub and would require an ADR before activation.
- **Confidence (ADR survives):** 🟢 High — per-platform hookspecs are correct given the platform asymmetry documented in ADR-0078.

---

### Assumption 3.3: The Ack-First Contract Covers All Interactive Platform Events

- **Stated Norm:** "For all interactive platform events that require an explicit acknowledgment, `ack()` MUST be called before any domain logic executes." (Standard 6)
- **Challenge:** Standard 6 F4 says "if `ack()` fails, do NOT proceed to domain logic." But what if domain logic is needed to compute the ack payload? For example, a Slack view submission that requires a synchronous ack with a validation error (before the form closes). In this case, the domain service must run to determine if the input is valid — before `ack()`.
- **Evidence Strength:** ⭐⭐⭐ Strong — Slack Bolt supports `ack()` with an error payload: `await ack({"response_action": "errors", "errors": {...}})`. Computing those errors may require domain service calls. Standard 6 F1 says `ack()` is "unconditional" — this appears to preclude ack-with-error.
- **Counter-Evidence:** The challenge is valid for the specific case of *form validation errors returned synchronously in the ack*. However, Standard 6's "unconditional" is about the timing (ack always comes before domain work) not the payload. An `ack()` with an error response is still called first (before domain I/O writes); the input validation is a synchronous check that does not write domain state. This is consistent with Standard 8 S4 ("illegal transitions raise a domain exception; `ingress.py` maps this to `OperationResult.permanent_error`"). The `ack()` with error payload is the Slack adapter's translation of that `permanent_error`.
- **Confidence (ADR survives):** 🟢 High — the ack-first contract is correct. The ack-with-error case is a natural extension of F1 that is implicit in the permanent_error mapping. A clarifying note in F1 or F4 would strengthen the standard but is not normatively required.

---

### Assumption 3.4: NormalisedIntent.raw_payload Decouples ingress.py from Platform Details

- **Stated Norm:** "N2: `ingress.py` must not access `raw_payload` for routing decisions. Routing is based solely on `intent`." (Standard 7 N2) "`raw_payload` is passed to the domain service only if the service requires platform-specific context it cannot fetch from DynamoDB."
- **Challenge:** If `raw_payload` can be passed to `service.py`, then `service.py` indirectly receives platform-specific data — violating the principle that `service.py` is channel-agnostic. Over time, `service.py` methods may accumulate `raw_payload` key-lookups that create hidden platform coupling.
- **Evidence Strength:** ⭐⭐ Moderate — this is an incremental accumulation risk.
- **Counter-Evidence:** Standard 1 A4 prevents `service.py` from importing platform SDK types. N2's qualifier "only if the service requires platform-specific context it cannot fetch from DynamoDB" is meant to be exceptional, not a regular pattern. The `raw_payload: dict` opaque type makes the coupling visible at the type level. Standard 7 N2's qualifier should be interpreted narrowly. A code review standard (S8.1-style enforcement) for `raw_payload` key access in `service.py` would be a useful complement, but the ADR's principle is sound.
- **Confidence (ADR survives):** 🟢 High — the principle is sound; the risk is managed by Standard 1 A4's import restriction.

## 4. Failure Modes Identified

### Failure Mode 4.1: ingress.py Lock-Gating Code Not Migrated (Pre-Condition Verified — CONFIRMED)

- **ADR Statement:** "Existing handler code that currently calls `ingress.py` as a lock-gating admission step must be migrated to the domain service before this ADR is marked Accepted." (Migration section)
- **Verification result (2026-05-07):** `app/packages/access/sync/interactions/ingress.py` was inspected. Both `enqueue_user_sync` and `enqueue_platform_sync` import and call `check_lock` from `packages.access.sync.platform_lock`, performing lock-gating directly inside ingress. The migration to `service.py` has **not** been done. This is a confirmed code work item blocking ADR-0089 acceptance.
- **Risk:** The current code violates Standard 5 I3 (ingress must not perform lock-gating; that is a domain service concern). Accepting ADR-0089 before the migration is complete would create a governance compliance gap.
- **Probability:** Confirmed present — not a risk estimate.
- **Mitigation:** Code migration required before marking Accepted. See Follow-Up Actions §9.

---

### Failure Mode 4.2: SQS Enqueue Failure After Domain Write (Standard 9 E2 Gap)

- **Risk:** Standard 9 E2 says "SQS enqueues must happen ONLY after the domain entity write succeeds." But what happens if the entity write succeeds and the SQS enqueue fails for a non-critical notification? The interaction is not stalled (the entity is in a valid state) but the notification is never delivered.
- **Coverage:** Standard 9 (compensation table) addresses this: "Platform notification → Enqueue to SQS notification queue after entity write. SQS provides at-least-once delivery with DLQ." The DLQ policy (ADR-0091 Standard 6) governs undeliverable SQS messages. However, if the enqueue itself fails before reaching SQS (transient network error), there is a lost notification. Standard 9's "critical gate" class and transactional outbox are the governed mitigation for this case. E5 states classification as "critical gate" is a feature-level decision.
- **Assessment:** Adequately governed. The responsibility for classifying critical vs. non-critical notifications is correctly delegated to Tier-4 feature ADRs (Standard 9 E5).

## 7. Tradeoffs Accepted

| Tradeoff | Rationale |
|----------|-----------|
| Ack-first model creates a gap between ack success and domain write success | Mandatory per Slack/Teams platform contracts. The partial failure compensation framework (Standard 9) is the mitigation. |
| `NormalisedIntent.raw_payload: dict` is untyped | Typed variants would require per-feature payload types — reducing the generality of `ingress.py`. The opaque `dict` is intentional. |
| `ingress.py` cannot perform lock checks | Lock checks moved to `service.py` (Standard 5). This creates a migration obligation for existing code. Tradeoff: cleaner separation of concerns vs. migration cost. |
| Per-platform hookspecs require code change for each new platform | Intentional: platform APIs are asymmetric (ADR-0078). Generic registration would lose type safety at platform service boundaries. |
