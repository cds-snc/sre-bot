## 3. Assumptions Challenged

### Assumption 1: One request has exactly one transport context

**Claim:** A single in-flight request is always within one transport context — it is either an HTTP JWT request, or a Slack command, or a webhook, or a system call. There is no case where a single request needs to resolve identity from multiple transports simultaneously.

**Challenge Method:** Enumerate concrete request paths in the application:

- `GET /api/v1/...` with `Authorization: Bearer <JWT>` → HTTP transport, JWT path only.
- Slack `/command` → Slack Bolt handler over Socket Mode WebSocket, Slack `user_id` path only. Note: this is NOT an HTTP request; trust is established at WebSocket connection level via `SLACK_APP_TOKEN`, not per-message HMAC signing.
- `POST /hook/{webhook_id}` → webhook middleware, HMAC + payload path only.
- Background job (system actor) → no inbound request, system constant.

**Result:** No counterexample found. All four paths are mutually exclusive per request. Multi-source orchestration within a single request is not a valid use case in this application.

**Confidence:** 🟢 High

---

### Assumption 2: `resolve_from_jwt` has no external API call

**Claim:** JWT claim extraction from a pre-validated payload is a pure local transform. The JWKS-based signature validation happens upstream in `JWKSManager` before the payload reaches `IdentityService`. `resolve_from_jwt` does not call any external API.

**Challenge Method:** Read `app/infrastructure/identity/resolver.py` `resolve_from_jwt` method. Inspect all calls made.

**Result:** `resolve_from_jwt` in the resolver extracts `email`, `sub`, `name` etc. from the `jwt_payload` dict. It creates a `User` object from these claims. No HTTP call, no SDK call, no I/O. The JWKS signature verification is performed by `JWKSManager.decode_token()` before the payload is passed to the resolver.

**Confidence:** 🟢 High

---

### Assumption 3: Narrowing the Protocol does not break any current caller

**Claim:** The only production caller of `resolve_from_jwt` is `app/infrastructure/security/current_user.py`. The other methods (`resolve_from_slack`, `resolve_from_webhook`, `resolve_system_identity`) have no callers outside the identity package itself.

**Challenge Method:** Search codebase for usages of `resolve_from_slack`, `resolve_from_webhook`, `resolve_system_identity` in feature packages and routes.

**Result:** Grep search confirms no imports of these method names from feature packages or routes. They exist only within `app/infrastructure/identity/` (resolver and service). Removing them from the Protocol affects no feature caller.

**Confidence:** 🟢 High

---

### Assumption 4: `SlackPlatformProvider` owning Slack user resolution is consistent with ADR-0078

**Claim:** ADR-0078 governs concrete platform providers. Moving Slack user ID → `User` resolution into `SlackPlatformProvider` as Category C is consistent with ADR-0078's pattern for platform-specific operations.

**Challenge Method:** Review ADR-0078 classification model. Category C = implementation detail, no Protocol required, owned by platform layer.

**Result:** ADR-0078 Standard 1 explicitly states platform providers are concrete classes (no Protocol), Category C. User resolution from a Slack `user_id` is inherently Slack-specific (requires `users.info` API call), will never be substituted across platforms, and does not need a Protocol. ADR-0078 Category C is the correct classification.

**Confidence:** 🟢 High

---

### Assumption 5: The amendment does not create a gap in the identity resolution story

**Claim:** After the amendment, every identity resolution path has a defined owner: JWT → `IdentityService`, Slack → `SlackPlatformProvider` (ADR-0067 S8), Webhook → middleware, System → inline. No path is orphaned.

**Challenge Method:** Map each identity source to its post-amendment owner.

**Result:**

| Identity Source | Owner | Governed By |
|-----------------|-------|-------------|
| JWT Bearer token | `IdentityService` Protocol | ADR-0061 S3 |
| Slack `user_id` | `SlackPlatformProvider` (Category C) | ADR-0067 S8, ADR-0078 |
| Webhook actor | Webhook middleware (inline) | ADR-0061 S2 |
| System actor | Inline constant at call site | ADR-0061 S2 |

No gap. Every source has exactly one governed owner.

**Confidence:** 🟢 High
