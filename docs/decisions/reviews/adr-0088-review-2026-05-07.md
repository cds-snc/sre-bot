  three-layer outbound model (Standards 7–8), which remains uniquely ADR-0088's scope.

## 3. Resolution of Round 1 Primary Blockers

### Blocker 1: S5.2 Callable Signature vs. Runtime Arity Dispatch

**Round 1 Finding:** ADR-0088 S5.2 specified `Callable[[CommandPayload], CommandResponse]`
but `base.py` dispatches handlers with 1, 2, or 3 positional arguments based on runtime
conditions. The type safety claim was inaccurate.

**Round 2 Status: ✅ RESOLVED (architectural repositioning)**

ADR-0089 Standard 7 introduces `NormalisedIntent` as a `TypedDict` — the canonical
adapter-to-ingress boundary type. `ingress.py` consumes `NormalisedIntent`, not
`CommandPayload`. The arity-dispatch problem in `base.py` is now scoped entirely to the
platform provider internals (`app/infrastructure/platforms/`), which is an infrastructure
implementation concern — not a feature ingress contract.

ADR-0088 S5's framing of `CommandPayload` as the inbound feature boundary type is superseded
by ADR-0089 Standard 7. If `CommandPayload`/`CommandResponse` are retained at all, they are
provider-internal types and should be described as such, not as "the governed boundary types
for all platform command handlers" visible to feature packages.

**Required action for ADR-0088:** Remove or redirect S5 to ADR-0089 Standard 7. If
`CommandPayload` is documented, scope it explicitly to `app/infrastructure/platforms/`
internals, not to feature-package ingress contracts.

---

### Blocker 2: `CommandPayload` Not `frozen=True` (ADR-0065 Violation)

**Round 1 Finding:** `CommandPayload` was a plain `@dataclass` with a `__post_init__`
self-assignment, making it incompatible with `frozen=True`. ADR-0065 requires `frozen=True`
for internal domain data. ADR-0088 did not declare the deviation.

**Round 2 Status: ✅ RESOLVED (architectural repositioning)**

ADR-0089 Standard 7 defines `NormalisedIntent` as a `TypedDict`. Per the ADR-0065 type
boundary table, `TypedDict` is the correct type for "dict-shaped adapters." The NormalisedIntent
is not a domain model; it is a normalised transport-to-ingress projection. The ADR-0065
`frozen=True` requirement applies to domain entities (`@dataclass(frozen=True)`) — not to
TypedDict adapter shapes. There is no deviation.

`CommandPayload`, repositioned as a platform-provider internal implementation type, falls
outside the feature domain model boundary and is no longer subject to the ADR-0065 "internal
domain data" row. The `__post_init__` issue becomes a provider implementation detail to resolve
independently (noted in the code quality follow-up from Round 1).

**Required action for ADR-0088:** Acknowledge that the ingress boundary type is `NormalisedIntent`
(TypedDict, ADR-0089 Standard 7) and that `CommandPayload` is a provider-internal type — not
part of ADR-0088's governed boundary surface.

## 4. New Findings from ADR-0089/0090/0091 Overlap Analysis

### Finding 4.1: Standards 1–6 Superseded by ADR-0089 (Major)

ADR-0089 covers every inbound handler concern addressed by ADR-0088 Standards 1–6, with
amendments and greater precision:

| ADR-0088 Standard | ADR-0089 Coverage | Divergence |
|-------------------|-------------------|------------|
| S1: Transport-Agnostic Ingress Layer | Standard 5 (`ingress.py` contract — intent resolver + idempotency gate) | ADR-0089 S5 is more precise: ingress does not perform admission control or lock checks (those belong in `service.py`). ADR-0088 S1 was aligned with ADR-0059's "admission logic" framing, which ADR-0089 explicitly corrects. |
| S2: Adapter Independence | Standards 1 (A1–A7) + Standard 2 (B1–B6) | Fully absorbed; ADR-0089 adds rules A2–A5 not present in ADR-0088 S2 |
| S3: Per-Platform Hookspec Registration | Standard 3 (K1–K5) | **Hookspec names changed:** `register_slack_commands` → `register_slack_interactions`; `register_teams_commands` → `register_teams_interactions`; new `register_event_handlers` hookspec added (blinker ADR-0083). ADR-0088 S3 references the old names, making it divergent from the canonical registration surface. |
| S4: `interactions/` Directory Contract | Standard 2 (directory structure) | Substantially identical; ADR-0089 adds `presenters.py` and `adapters/` to the directory tree. |
| S5: Typed Command Handler Contract | Standard 7 (`NormalisedIntent` TypedDict) | Architectural model changed — see Blocker 1 resolution above |
| S6: Error Mapping Per Transport | Standard 8 Step 6 (return platform response); ADR-0050 via Standard 5 I4 | ADR-0089 S8 Step 6 handles this; ADR-0088 S6 is redundant |

**Consequence:** A developer reading ADR-0088 Standards 1–6 will encounter inbound handler
rules that diverge from the ADR-0089 canonical model:

- Wrong ingress description (`ingress.py` as "shared admission logic" vs. "intent resolver")
- Wrong hookspec names (`register_slack_commands` vs. `register_slack_interactions`)
- Wrong boundary type at ingress (`CommandPayload` vs. `NormalisedIntent`)

This creates a three-way conflict (ADR-0088 S1–S6, ADR-0089, ADR-0059) for developers
navigating inbound handler architecture.

**Required action for ADR-0088:** Remove Standards 1–6 from ADR-0088 and replace with a
single reference block: *"Inbound handler standards are governed by ADR-0089. Standards 1–6
of ADR-0088 are superseded by ADR-0089 in full."* Standards 7–8 (outbound model) remain.

---

### Finding 4.2: Standards 7–8 Remain ADR-0088's Unique Contribution (Positive)

The three-layer outbound platform model (Standard 7) and named role enforcement (Standard 8)
are **not covered by ADR-0089, ADR-0090, or ADR-0091**, and are not addressed in the
target-state architecture reference. They remain ADR-0088's sole unique scope.

ADR-0089 Standard 2 includes `adapters/` in the package directory tree as the outbound
integration directory. ADR-0088 Standards 7–8 are the authoritative governance for what lives
in `adapters/`: the three-layer decomposition (Intent → Realization → Transport), the
Reconciler vs. ClientAdapter naming convention, and the conditions under which the Reconciler
layer is required vs. optional.

**Consequence:** Standards 7–8 are actively needed. The existing `AwsIdentityCenterAdapter`
mixed-responsibility class remains pre-compliant with Standard 7, and the migration concern
from Round 1 Scenario 6.2 is still unaddressed.

**Required action for ADR-0088:** Retain Standards 7–8 verbatim. No changes needed to these
standards — they survive the ADR-0089/0090/0091 landscape intact.

### Finding 4.5: Target-State Alignment of Standards 7–8 (Positive)

The target-state architecture reference (`tmp/target-state-architecture-stateless-2026-05-06.md`)
is fully consistent with ADR-0088 Standards 7–8:

- Section 3.1 (Slack Service) and 3.2 (Teams Service) confirm standalone infrastructure
  services — consistent with the Intent → Realization → Transport separation where Reconciler
  and ClientAdapter are outbound feature concerns, not platform service concerns.
- The `adapters/` directory in ADR-0089's package structure (Section 3.2) is the canonical
  home for Reconciler + ClientAdapter pairs.
- The target-state architecture does not introduce any abstraction that conflicts with the
  three-layer outbound model or the Reconciler/ClientAdapter naming convention.

No revisions to Standards 7–8 are required on account of the target-state reference.

---

### Finding 4.6: Alternatives Considered Section Needs Minor Pruning (Minor)

The current Alternatives Considered section's Alternative 5 references "message bus dispatch"
as over-engineered for a "request-response" codebase. ADR-0089 Standard 9 and the target-state
reference establish that the codebase IS event-driven (blinker for in-process, SQS for
cross-process). The characterization "primarily request-response, not event-driven" is now
inaccurate and could mislead future readers about the architectural intent.

**Required action for ADR-0088:** Update Alternative 5's rationale to reflect the correct
framing: blinker and SQS are the event-driven mechanisms for in-process and async continuation
respectively; a unified message bus is still unnecessary because blinker + SQS cover the
distinct in-process vs. cross-process cases at lower complexity.

## 6. Revised Failure Mode Assessment

### Failure Mode 6.1: Developer Conflates ADR-0088 S1–S6 with ADR-0089

**Risk:** A developer reads ADR-0088 as the inbound handler standard and implements:

- `ingress.py` as an admission-control gate (ADR-0088 S1 framing) instead of an intent
  resolver + idempotency gate (ADR-0089 S5 framing)
- `register_slack_commands` hookimpl (ADR-0088 S3) instead of `register_slack_interactions`
  (ADR-0089 S3 K1) — the hookspec never fires for the feature's Slack handlers
- `CommandPayload` as the ingress boundary type instead of `NormalisedIntent`

**Probability:** High — ADR-0088 and ADR-0089 are in the same Wave 7 suite; both are Draft;
no supersession relationship is currently declared between them.

**Mitigation:** Revise ADR-0088 to remove Standards 1–6 and add an explicit forward reference
to ADR-0089. This eliminates the ambiguity entirely.

---

### Failure Mode 6.2: `AwsIdentityCenterAdapter` Refactor Missed

**Risk (unchanged from Round 1):** Standards 7–8 mandate splitting mixed-responsibility
platform adapters. `AwsIdentityCenterAdapter` is pre-compliant. Without an active migration
item, this refactor is deferred indefinitely.

**Mitigation:** Implementation Guidance #4 (from ADR-0088) remains the correct vehicle. No
change to Standard 7 is required; the migration must be tracked separately.

## 7. Recommended Revision Path

ADR-0088 requires a **scope-narrowing revision**. The revision is well-defined and should be
low-effort relative to a full rewrite.

### Step 1: Collapse Standards 1–6 into a forward reference block

Replace Standards 1–6 with:

```markdown
### Standards 1–6: Superseded by ADR-0089

Inbound multi-transport handler architecture — including the transport-agnostic ingress
layer, adapter independence, per-platform hookspec registration, `interactions/` directory
contract, normalised intent boundary type, and error mapping per transport — is governed
by **ADR-0089 (Platform Interaction Handler Standard)**.

Note: the hookspec names in force are `register_slack_interactions`,
`register_teams_interactions`, `register_routes`, and `register_event_handlers` per
ADR-0089 Standard 3. The earlier names (`register_slack_commands`,
`register_teams_commands`) are not the current canonical forms.
```

### Step 2: Retain Standards 7–8 verbatim

No changes needed. Standards 7–8 content is architecturally correct, uniquely scoped,
and consistent with the target-state architecture.

### Step 3: Update ADR metadata

- `constrained_by`: replace `ADR-0059` with `ADR-0089`; add `ADR-0090`, `ADR-0091`, `ADR-0050`
- `related_records`: add `ADR-0089`, `ADR-0090`, `ADR-0091`
- `impacts`: add `ADR-0089` (it absorbs Standards 1–6; bidirectional cross-reference needed)

### Step 4: Correct the context table example name

Replace `AwsIdentityStoreReconciler` with `AwsIdentityCenterAdapter (current) → AwsIdentityStoreReconciler (proposed)` or equivalent explicit wording.

### Step 5: Update Alternatives Considered Alternative 5

Revise the "request-response, not event-driven" framing to: "blinker (in-process) and SQS
(cross-process async) are the governed event-driven mechanisms per ADR-0083 and ADR-0079.
A unified message bus adds complexity without covering cases not already served by blinker
and SQS."

### Step 6: Update Compliance and Boundaries section

Under "This ADR governs," retain only Standards 7–8 topics. Under "This ADR does not govern,"
add: "Inbound multi-transport handler architecture (governed by ADR-0089)."

## 9. Binary Gate Outcome

**GATE DECISION:**

🟡 **REVISE (scope narrowing)** → ADR-0088 requires revision before it can proceed to
Accepted. The revision is well-scoped and low-risk.

**Primary Blockers:**

1. **Standards 1–6 compete with ADR-0089:** ADR-0088 Standards 1–6 cover the same ground
   as ADR-0089 with divergent details (wrong hookspec names, wrong boundary type at ingress,
   outdated ingress role description). A developer following ADR-0088's inbound standards will
   produce non-compliant code. Standards 1–6 must be replaced with a forward reference block
   before ADR-0088 can be accepted.

2. **`constrained_by` references superseded ADR-0059:** ADR-0059 was superseded by ADR-0089.
   The constraint chain must be updated to reflect the live ADR hierarchy.

**Remaining scope after revision:** ADR-0088 becomes a focused Tier-2 standard governing
outbound platform interaction design — the three-layer Intent → Realization → Transport
model and named role enforcement. This is a coherent, uniquely-scoped ADR with no coverage
overlap with any other ADR in the 0085–0091 suite.
