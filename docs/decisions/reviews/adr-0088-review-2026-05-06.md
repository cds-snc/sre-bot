**Finding 2A-1: S5.2 Callable Signature vs. Runtime Dispatch (Major)**

ADR-0088 S5.2 specifies `Callable[[CommandPayload], CommandResponse]` as the command handler
signature. Inspection of `app/infrastructure/platforms/providers/base.py` lines 447–451 reveals:

```python
# Handler expects: payload, parsed_args, request
response = cmd_def.handler(payload, parsed_args, request)
elif parsed_args is not None:
    # Handler expects: payload, parsed_args
    response = cmd_def.handler(payload, parsed_args)
else:
    # Handler expects: payload only
    response = cmd_def.handler(payload)
```

The provider dispatches with 1, 2, or 3 positional arguments based on runtime conditions.
The actual `geolocate` handler uses `handle_geolocate_command(payload: CommandPayload, parsed_args: Dict[str, Any]) -> CommandResponse`.
The access/sync `handle_sync_command` uses `(payload: CommandPayload) -> CommandResponse` (no parsed args).
The platform provider infers arity from whether `cmd_def.arguments` is populated — not from the callable's type signature.

This runtime arity dispatch is type-unsafe (`cmd_def.handler` cannot be typed as `Callable[[CommandPayload], CommandResponse]` when the provider calls it with 2 or 3 args). ADR-0088 S5.2 must document all valid handler signatures or re-specify the contract to reflect the actual injection model.

**Finding 2A-2: `CommandPayload` Not `frozen=True` (Major)**

`CommandPayload` in `infrastructure/platforms/models.py` uses plain `@dataclass` with a
`__post_init__` that assigns `self.correlation_id = f"cmd-{datetime.utcnow().timestamp()}"`.
ADR-0065 specifies `@dataclass(frozen=True)` for internal domain data. `CommandPayload` is an
internal domain boundary type (not HTTP/webhook I/O at a FastAPI boundary), so ADR-0065's rule
applies. The `__post_init__` self-assignment makes `CommandPayload` structurally incompatible
with `frozen=True` without refactoring to a factory function or `object.__setattr__` pattern.

ADR-0088 Standard 5 must either: (a) require `frozen=True` and flag the `__post_init__` refactor
as a follow-up action, or (b) declare and justify the ADR-0065 deviation.

## 3. Assumptions Challenged

### Assumption 3.1: Transport-Agnostic Ingress Layer Is the Correct Locus for Shared Logic

- **Stated Norm:** "Every feature operation reachable from more than one transport must have a single
  transport-agnostic ingress function. Transport adapters call this function; they do not re-implement
  business logic." (Standard 1)
- **Underlying Assumption:** The set of operations that must be shared across transports is stable
  and identifiable at design time. The ingress function can be written without transport-specific
  dependencies.
- **Challenge:** Real admission logic (enabled-checks, lock-checks, job spawns as in `ingress.py`)
  is not "pure business orchestration" as ADR-0088 describes it — it is admission control. ADR-0059
  B3 accurately calls it "shared admission logic (feature-enable checks, concurrency guards)." The
  naming tension is minor but creates interpretation drift: developers reading ADR-0088 may exclude
  admission control from `ingress.py` because "business orchestration" implies richer logic.
  The actual `enqueue_user_sync()` in `access/sync/interactions/ingress.py` performs feature flag
  check + lock check + job spawn — closer to ADR-0059's "admission" than ADR-0088's "orchestration."
- **Evidence Strength:** ⭐ Strong (code observed directly)
- **Counter-Evidence Found:** Yes — `ingress.py` content matches ADR-0059's characterization more
  than ADR-0088's
- **Confidence (ADR survives challenge):** 🟢 High — the architectural pattern is correct regardless
  of terminology. The fix is a wording alignment with ADR-0059, not a structural revision.
- **Reviewer Notes:** ADR-0088 should align its description of `ingress.py` with ADR-0059 B3's
  phrasing. "Transport-agnostic shared admission and orchestration logic" covers both cases.

---

### Assumption 3.2: Per-Platform Hookspec Model Provides Type Safety

- **Stated Norm:** "Each hookspec provides the platform-specific provider type as argument...
  Manageable scale: ≤6 transports is within the range where explicit hookspecs are clearer than
  a generic registry." (Standard 3 Rationale)
- **Underlying Assumption:** Type safety is preserved because each hookspec carries a typed
  platform provider; features receive the correct concrete type without `Callable[..., Any]` erasure.
- **Challenge:** The hookspec model provides type safety at the registration call site
  (the `@hookimpl` receives `SlackPlatformProvider`). But within the provider, command handler
  dispatch uses runtime arity inspection (Finding 2A-1) which bypasses mypy enforcement. The
  hookspec-level type safety does not propagate through to the handler invocation — the provider
  calls `cmd_def.handler(payload, parsed_args)` where `handler` is typed as `Optional[Callable[..., CommandResponse]]`. The actual type safety claimed by S5.2 is not delivered by the current implementation.
- **Evidence Strength:** ⭐ Strong (confirmed in `base.py` lines 447–451)
- **Counter-Evidence Found:** Yes — `base.py` runtime dispatch is type-unsafe despite ADR's claim
- **Confidence (ADR survives challenge):** 🟡 Moderate — the hookspec model is correct and valuable;
  the type safety gap is in S5.2's handler callable specification, not in the hookspec design itself.
  The ADR survives if S5.2 is revised to accurately document the handler invocation contract.
- **Reviewer Notes:** Two paths to resolution: (a) specify all three valid handler signatures and
  note that typed dispatch relies on the provider's injection convention, or (b) propose a
  typed `CommandHandler` Protocol with multiple overloads or a typed injection mechanism.

---

### Assumption 3.3: Three-Layer Outbound Model Is Necessary for Complex Outbound Interactions

- **Stated Norm:** "When a feature interacts with an external platform to realize intent (not just
  dispatch commands), the outbound interaction must follow a three-layer model separating concerns."
  (Standard 7)
- **Underlying Assumption:** The distinction between Realization (reconciliation planning) and
  Transport (protocol translation) is clean enough in practice to justify separate classes. Mixed-
  responsibility classes will accumulate decision logic.
- **Challenge 1:** The `AwsIdentityCenterAdapter` in `access/sync/adapters/` already exists as
  a mixed-responsibility class: it declares `capabilities()` (Realization role), resolves entity
  IDs via `_resolve_group_id()` and `canonicalize_entitlement_id()` (entity matching, Realization),
  AND makes direct `self._aws.identitystore.*` calls (Transport). ADR-0088 Standard 8.1 would
  flag this class as a code review red flag. The ADR is self-consistent: it identifies the problem
  and mandates the fix via Implementation Guidance #4 ("Audit existing platform classes"). But the
  example in the Context table uses `AwsIdentityStoreReconciler` — a name that does not exist in
  the codebase; the actual class is `AwsIdentityCenterAdapter`. The example conflates the desired
  post-refactor name with the current class, potentially confusing implementors.
- **Challenge 2 (over-engineering for simple cases):** S7.5 explicitly exempts thin cases:
  "For features with only thin API translation and no reconciliation logic (e.g., simple notification
  dispatch), the Reconciler layer may be omitted." The scenario table confirms. The three-layer
  model is not universally mandatory — it is scoped to reconciliation-heavy outbound interactions.
  Over-engineering concern does not materialize.
- **Evidence Strength:** ⭐ Strong (code observed; ADR is accurate about the existing problem)
- **Counter-Evidence Found:** Partial — the existing class validates the problem but the example
  name is inaccurate
- **Confidence (ADR survives challenge):** 🟢 High — the three-layer model is architecturally sound
  and properly scoped. The name mismatch in the context table is a Minor precision issue.
- **Reviewer Notes:** The Context table should reference `AwsIdentityCenterAdapter` (the actual
  class) rather than `AwsIdentityStoreReconciler` (the proposed post-refactor name). Using the
  post-refactor name in an illustrative example without marking it as "proposed" is imprecise.

### Assumption 3.4: `interactions/` Directory as the Mandatory Contract Surface for Multi-Transport

- **Stated Norm:** "The `interactions/` directory becomes mandatory when a second transport is
  added." (S4.3) The table in Context shows `geolocate` has no `interactions/` directory
  ("❌ No (trivially thin)").
- **Underlying Assumption:** The `interactions/` directory contract is the only valid pattern for
  multi-transport features; `platforms/` or other directory names used by existing features are
  migration debts, not alternative patterns.
- **Challenge:** `geolocate` currently has `packages/geolocate/platforms/slack.py`,
  `platforms/teams.py` (stub), and `platforms/discord.py` (stub) — it IS multi-transport but uses
  `platforms/` not `interactions/`. The ADR Context table marks geolocate as "❌ No (trivially thin)"
  which understates the reality: geolocate HAS a Slack handler and the `platforms/` directory.
  ADR-0059 Standard 2 already mandates `interactions/`; ADR-0088 reinforces it. The geolocate
  `platforms/` directory is non-compliant with both ADRs today, but Implementation Guidance #3
  correctly identifies the migration path: "Geolocate migration: Current `platforms/` directory
  should be renamed to `interactions/`." The ADR is accurate that geolocate's thin handler doesn't
  require a separate `ingress.py` (S1.4 exception applies), but the directory name must change.
- **Evidence Strength:** ⭐ Strong (confirmed via directory listing)
- **Counter-Evidence Found:** Yes — the Context table's characterization of geolocate as lacking
  an ingress layer ("❌ No (trivially thin)") elides the fact that geolocate uses `platforms/` not
  `interactions/`. This is mildly misleading: the table addresses ingress layer presence, not
  directory naming compliance.
- **Confidence (ADR survives challenge):** 🟢 High — the ADR's rule is correct and its
  Implementation Guidance acknowledges the migration need. The Context table's characterization
  is imprecise but not wrong regarding ingress layer applicability.
- **Reviewer Notes:** The Context table for `geolocate` should note the directory rename migration
  need, not merely the ingress layer exception.

## 4. Failure Modes Identified

### Failure Mode 4.1: S5.2 Callable Signature → Type-Unsafe Handler Dispatch

- **If Assumption Fails:** Handler callable signatures (1, 2, or 3 args) are not enforced by mypy
  because `handler` is `Optional[Callable[..., CommandResponse]]`. A feature developer following
  ADR-0088 S5.2 writes `handler(payload: CommandPayload) -> CommandResponse`. If `cmd_def.arguments`
  is populated, the provider calls `handler(payload, parsed_args)` — Python raises `TypeError` at
  runtime (unexpected positional argument).
- **Platform Impact:**
  - Incident management workflow: Low (incident commands are simple, unlikely to need parsed_args)
  - Access synchronization workflow: Low (access sync uses no `parsed_args` in its handler)
  - Access request workflow: Medium (request commands may use arguments)
  - Multi-provider integrations (Slack, Teams, GWS, AWS, GitHub): Medium (any Slack command using
    argument parsing hits this gap if developer follows S5.2 literally)
- **Probability Estimate:** Medium — any new Slack command with argument parsing that follows S5.2
  literally will encounter this. Existing handlers work because they were written before S5.2.
- **Mitigation or Acceptance:** Revise S5.2 to document all valid handler arities. Until then,
  developers must consult `base.py` directly to understand the dispatch convention — defeating the
  purpose of having a standard.

---

### Failure Mode 4.2: `CommandPayload` Not Frozen → Aliasing / Mutation Bugs

- **If Assumption Fails:** A platform adapter (or test stub) receives a `CommandPayload`, mutates a
  field, and inadvertently affects downstream processing. Plain `@dataclass` with `Dict[str, Any]`
  fields allows both field reassignment and in-place mutation of `platform_metadata`. For a shared
  object passed across adapter and ingress layers, this creates latent aliasing bugs.
- **Platform Impact:**
  - Incident management workflow: Low (no shared payload mutation observed)
  - Access synchronization workflow: Low (current implementation doesn't mutate payload)
  - Access request workflow: Low
  - Multi-provider integrations: Low — current risk is latent, not active
- **Probability Estimate:** Low currently; Medium as codebase grows and `platform_metadata` is used
  more extensively.
- **Mitigation or Acceptance:** Adopt `frozen=True` plus a factory function for correlation ID
  generation (e.g., `CommandPayload.create(...)`), or use `object.__setattr__` in `__post_init__`.
  The risk is latent today but the ADR-0065 alignment gap should be closed proactively.

---

### Failure Mode 4.3: Ingress Terminology Drift

- **If Assumption Fails:** Developers read ADR-0088's "pure business orchestration" and conclude that
  admission control (feature flag, lock check) belongs in the HTTP route handler or service layer
  rather than `ingress.py`, duplicating it per transport. This inverts the pattern the ADR intends.
- **Platform Impact:**
  - All workflows using multi-transport features: Medium — duplicated enabled-checks and lock-checks
    would create transport-specific divergence exactly as the ADR seeks to prevent.
- **Probability Estimate:** Low — ADR-0059 B3 uses the correct "admission logic" terminology, and
  developers consulting ADR-0059 get the accurate framing. But ADR-0088 is the canonical dispatch
  standard and its terminology should not contradict ADR-0059.
- **Mitigation or Acceptance:** Align ADR-0088 Standard 1 description language with ADR-0059 B3.

## 7. Tradeoffs Accepted

### Tradeoff 7.1: Per-Platform Hookspecs vs. Unified Platform Registry

- **Chosen:** Per-platform hookspecs (`register_slack_commands`, `register_teams_commands`)
- **Rejected:** Single `register_platform_commands(platform: str, provider: Any)` with
  string-keyed dispatch
- **Rationale:** Platform handler signatures are fundamentally asymmetric (Bolt middleware vs.
  Bot Framework TurnContext). A unified registry would require `Callable[..., Any]` type erasure.
  Four explicit hookspecs are clearer and type-safe at the registration site.
- **Risk Accepted:** Linear scaling of hookimpls (features × platforms). At 6 features × 4 platforms
  = 24 hookimpls; manageable but mechanical. Revisit threshold set at 6 platforms.
- **Contingency:** If platform count exceeds 6, introduce a typed discriminated union or Protocol
  hierarchy for platform providers.

---

### Tradeoff 7.2: Three-Layer Outbound vs. Single "Adapter" Class

- **Chosen:** Intent (Feature Service) → Realization (Reconciler) → Transport (ClientAdapter)
- **Rejected:** Everything-in-an-Adapter pattern where a single class handles planning + SDK calls
- **Rationale:** `AwsIdentityCenterAdapter` is empirical evidence that everything-in-an-adapter
  causes architectural drift. Naming mixed-responsibility classes generically ("Adapter") hides
  their actual role scope. The three-layer model makes responsibility visible at code-review time.
- **Risk Accepted:** Additional files per platform per feature for complex outbound operations.
  For simple outbound (notifications), S7.5 exempts the three-layer requirement.
- **Contingency:** S7.5 escape hatch for thin operations is the contingency.

---

### Tradeoff 7.3: Transport-Agnostic Ingress Layer vs. Direct Service Invocation

- **Chosen:** Mandatory `ingress.py` for multi-transport features; S1.4 exception for trivially thin
- **Rejected:** Each adapter invokes the service layer directly; orchestration duplicated per adapter
- **Rationale:** Cosmic Python Appendix B (entrypoints pattern), Hexagonal Architecture (shared port),
  and the access/sync implementation all validate that shared admission logic belongs in a single
  function called by all adapters. The S1.4 exception is well-scoped (single-call, no orchestration).
- **Risk Accepted:** Structural overhead for HTTP-only features that add a second transport later
  (they must refactor to add `ingress.py`). S4.3 defers this cost until a second transport arrives.
- **Contingency:** S4.3 explicitly defers the `interactions/` directory requirement — no sunk cost
  in HTTP-only features.

---

### Tradeoff 7.4: Named Role Enforcement vs. "Adapter" as Umbrella Term

- **Chosen:** `*Reconciler` / `*ClientAdapter` / `*Service` / `*Coordinator` role-specific naming
- **Rejected:** Generic `*Adapter` for all platform-facing classes regardless of responsibility
- **Rationale:** `*Adapter` as an umbrella term produces `AwsIdentityCenterAdapter` that hides
  diffing logic, capability planning, and entity matching alongside SDK calls. Code review cannot
  detect role boundary violations without reading the class body. Named role enforcement makes
  drift visible at the method-name / file-name level.
- **Risk Accepted:** Rename churn for legacy classes. S8.2 permits backward-compatible aliases
  during migration.
- **Contingency:** S8.2 alias pattern manages the backward-compatibility window.

## 9. Binary Gate Outcome

**GATE DECISION:**

🔴 **REVISE** → ADR-0088 requires authoring revision; return to Step 5-9 author team with feedback

**Primary Blockers:**

1. **S5.2 Callable Signature Inaccuracy:** Standard 5.2 specifies `Callable[[CommandPayload], CommandResponse]`
   as the command handler signature. The actual platform dispatch in `base.py` calls handlers with
   1, 2, or 3 positional arguments (payload; payload + parsed_args; payload + parsed_args + request)
   based on runtime conditions. ADR-0088's claim that per-platform hookspecs provide type safety is
   undermined by this untyped arity dispatch. Standard 5.2 must be revised to describe the full
   dispatch protocol or propose a typed multi-signature `CommandHandler` Protocol to replace the
   current runtime arity inspection.

2. **`CommandPayload` ADR-0065 Deviation Undeclared:** ADR-0065 mandates `@dataclass(frozen=True)`
   for internal domain data. `CommandPayload` and `CommandResponse` are plain `@dataclass`, and
   `CommandPayload.__post_init__` performs direct field assignment incompatible with frozen=True.
   ADR-0088 Standard 5 must either adopt `frozen=True` (with implementation follow-up to refactor
   `__post_init__`) or explicitly declare and justify the ADR-0065 deviation. Without this
   declaration, ADR-0088 silently violates a sibling Tier-2 standard.
