## 3. Assumptions Challenged

### Assumption 3.1: Socket Mode Is the Only Viable Transport

- **Stated Norm:** "Private VPC deployment without public ingress is a hard constraint. Socket Mode satisfies this without additional infrastructure."
- **Underlying Assumption:** The application will always run in a private VPC without public HTTPS ingress for Slack.
- **Challenge:** If the deployment model changes (e.g., public ALB added for other purposes, or Slack introduces VPC PrivateLink), Socket Mode may no longer be the only or best option.
- **Evidence Strength:** ⭐ Strong — current ECS/VPC architecture confirms private deployment; no public endpoints exist. Terraform confirms no public ALB listener for Slack.
- **Counter-Evidence Found:** No — Slack PrivateLink or VPC peering options do not exist today.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The `SLACK_SOCKET_MODE` field (S5 R5) provides an explicit escape hatch for future HTTP Events API mode, which adequately mitigates this risk.

### Assumption 3.2: Fail-Fast on Invalid Credentials Is Correct

- **Stated Norm:** "If `SLACK_ENABLED` is true but credentials are invalid, startup must fail fast. Slack is not a degraded-start service."
- **Underlying Assumption:** Slack is always a primary transport; running without it is never useful when explicitly enabled.
- **Challenge:** In staging or feature-branch environments, developers may want to test HTTP-only functionality without valid Slack credentials. Fail-fast prevents this.
- **Evidence Strength:** ⭐⭐ Moderate — depends on deployment environment configuration practices.
- **Counter-Evidence Found:** No — current practice uses `SLACK_ENABLED=false` in non-Slack environments, which bypasses this entirely (S1 R2).
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The escape hatch is `SLACK_ENABLED=false`. Developers who don't need Slack set the flag to false. Fail-fast applies only when Slack is explicitly enabled but misconfigured — correct behavior for production safety.

### Assumption 3.3: Single Connection per Process Is Sufficient

- **Stated Norm:** "Only one Socket Mode connection exists per process."
- **Underlying Assumption:** Message throughput from Slack will never exceed single-connection capacity, and graceful restarts are handled at ECS task level (rolling deployment), not connection level.
- **Challenge:** Slack documentation explicitly supports up to 10 connections for load balancing and graceful restart. If event volume increases, a single connection could bottleneck.
- **Evidence Strength:** ⭐⭐ Moderate — current volume is low (SRE team operations), but future scale is unknown.
- **Counter-Evidence Found:** Yes — Slack explicitly recommends multiple connections for high-throughput and graceful restarts.
- **Confidence (ADR survives challenge):** 🟡 Moderate
- **Reviewer Notes:** The single-connection design is appropriate for current scale. **The rationale must be corrected:** not "Slack enforces this" but "single connection chosen for operational simplicity; shutdown coordination with multiple connections adds complexity not justified at current scale." The ADR should acknowledge the 10-connection capability as a future scaling option.

### Assumption 3.4: 5-Second Shutdown Budget Is Sufficient

- **Stated Norm:** "Transport phase ≤5 s; thread join with timeout."
- **Underlying Assumption:** Slack's WebSocket close handshake completes within 5 seconds.
- **Challenge:** Network latency, Slack API slowness, or in-flight message acknowledgements could delay close beyond 5 seconds.
- **Evidence Strength:** ⭐ Strong — WebSocket close is typically sub-second; Bolt SDK handles reconnection failures gracefully.
- **Counter-Evidence Found:** No — no reports of Slack WebSocket close exceeding 5 seconds.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The timeout-based forced termination (S3 R2) with warning log is the correct mitigation. Daemon thread is terminated when process exits regardless.

### Assumption 3.5: Features Must Not Import Bolt SDK Types

- **Stated Norm:** "Features must not import or reference `SocketModeHandler`, `slack_bolt.App`, or any Bolt SDK type directly."
- **Underlying Assumption:** The provider's registration API can fully mediate all Slack interactions without features needing Bolt-native types.
- **Challenge:** Advanced Bolt features (custom middleware, `ack()` with response payload, view `private_metadata` parsing) may require Bolt-native types. The mediation layer could bottleneck feature development velocity.
- **Evidence Strength:** ⭐⭐ Moderate — depends on provider API comprehensiveness.
- **Counter-Evidence Found:** Partial — current codebase shows zero Bolt imports in `app/packages/` (compliant). `app/modules/` has direct Bolt access (frozen zone, expected).
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** Mediation layer is appropriate for SDK version isolation. If advanced Bolt features are needed, the provider's API should be extended — this is the intended design.

## 4. Failure Modes Identified

### Failure Mode 4.1: S2 R4 Rationale Error → Incorrect Architectural Constraint

- **If Assumption Fails:** A developer reads S2 R4 and believes Slack enforces single connection, then encounters Slack's 10-connection documentation. Trust in the ADR's accuracy is undermined.
- **Platform Impact:**
  - Incident management workflow: None
  - Access synchronization workflow: None
  - Access request workflow: None
  - Multi-provider integrations (Slack, Teams, GWS, AWS, GitHub): None
- **Probability Estimate:** High % (documentation is publicly accessible; any developer can find the contradiction)
- **Mitigation or Acceptance:** **REVISE** — correct the rationale to cite operational simplicity rather than platform enforcement.

### Failure Mode 4.2: `stop()` Implementation Gaps

- **If Assumption Fails:** `stop()` is called during shutdown and `handler.close()` raises an exception (e.g., already-closed WebSocket). The exception propagates to the lifespan handler, potentially disrupting the remaining shutdown sequence.
- **Platform Impact:**
  - Incident management workflow: Low (incident channels persist in Slack; missed shutdown cleanup has no data impact)
  - Access synchronization workflow: Low (sync jobs use DynamoDB locks with TTL; Slack shutdown failure doesn't affect lock lifecycle)
  - Access request workflow: Low
  - Multi-provider integrations (Slack, Teams, GWS, AWS, GitHub): Medium (other platform providers' shutdown may be disrupted if Slack's exception aborts the shutdown sequence)
- **Probability Estimate:** Low % (Bolt SDK's `close()` is generally well-behaved, but edge cases exist)
- **Mitigation or Acceptance:** The ADR correctly specifies exception-safe `stop()` (S3 R3, R4). Current implementation doesn't comply, but the ADR is the target-state norm. Implementation will fix this. **No ADR revision needed — this is an implementation gap, not an ADR gap.**

## 7. Tradeoffs Accepted

### Tradeoff 7.1: Fail-Fast vs. Degraded-Start for Slack

- **Chosen:** Fail-fast startup when `SLACK_ENABLED=true` but credentials invalid (S1 R3).
- **Rejected:** Degraded-start (run HTTP-only with Slack disabled at runtime).
- **Rationale:** Slack is the primary interaction transport. Running without it in production masks configuration errors and creates a silent partial outage — operators see "healthy" process but users can't interact.
- **Risk Accepted:** A misconfigured staging deploy will fail to start entirely, even if only HTTP testing is intended. Operators must set `SLACK_ENABLED=false` explicitly.
- **Contingency:** `SLACK_ENABLED=false` is the escape hatch. Zero-code-change mitigation.

### Tradeoff 7.2: Single Connection vs. Multi-Connection

- **Chosen:** Single Socket Mode connection per process (S2 R4).
- **Rejected:** Multiple connections for load balancing and graceful restart.
- **Rationale:** Operational simplicity — shutdown coordination, message ordering predictability, and handler dispatch determinism. Current event volume does not justify multi-connection complexity.
- **Risk Accepted:** If event throughput exceeds single-connection capacity, Slack interactions may lag. Single point of failure within a task (mitigated by Bolt SDK auto-reconnect and ECS multi-task deployment).
- **Contingency:** Slack supports up to 10 connections. Multi-connection can be added as a provider enhancement without ADR revision (implementation change only).

### Tradeoff 7.3: SDK Mediation vs. Direct Bolt Access

- **Chosen:** Features access Slack exclusively through `SlackPlatformProvider` methods; no direct Bolt SDK imports (S4 R5).
- **Rejected:** Allow features to import and use Bolt SDK types directly.
- **Rationale:** SDK version isolation — provider absorbs breaking changes in Bolt SDK upgrades. Features depend on stable provider API, not volatile SDK API.
- **Risk Accepted:** Provider API may lag behind Bolt SDK capabilities. Features needing advanced Bolt features must wait for provider API extension.
- **Contingency:** Provider API can be extended without ADR revision. If urgently needed, `provider.app` property (currently exposed) provides escape hatch for advanced Bolt usage.
