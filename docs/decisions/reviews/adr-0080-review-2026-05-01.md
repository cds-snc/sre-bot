## 3. Assumptions Challenged

### Assumption 3.1: The boundary between application and infrastructure is clear-cut

- **Stated Norm:** "Application architecture — the Python FastAPI codebase (`app/`)" vs "Hosting infrastructure — CI/CD pipelines, container orchestration…"
- **Underlying Assumption:** Every component can be cleanly classified into one of the two domains.
- **Challenge:** Some concerns straddle both domains. Configuration injection has an app-side contract ("env vars before process start") and an infra-side fulfillment ("ECS task definition secrets"). Health checks have app-side code (endpoint) and infra-side consumption (ALB target group). Where does the boundary fall for these?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No — the ADR explicitly addresses straddling concerns in Principle 2 with a contract table pattern. The table decomposes each straddling concern into app-side contract and infrastructure fulfillment.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The contract-based interface (Principle 2) is the standard approach from Hexagonal Architecture. Ports (app contracts) and Adapters (infra fulfillment) is a well-established decomposition pattern. The ADR correctly avoids claiming a binary classification and instead provides a decomposition mechanism.

### Assumption 3.2: Existing application ADRs don't need immediate amendment

- **Stated Norm:** "When an existing application ADR contains infrastructure-specific guidance, that guidance must be treated as informational context, not normative constraint."
- **Underlying Assumption:** Reclassifying existing infra-specific guidance as "informational" is sufficient without immediately rewriting affected ADRs.
- **Challenge:** Could a developer read ADR-0052's "Move release-phase configuration binding to the ECS task definition level" and treat it as a binding requirement, creating a dependency on ECS that ADR-0080 is designed to prevent?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No — the risk is acknowledged. ADR-0052's ECS-specific guidance is already classified as a "known implementation gap" and "explicit migration target" within ADR-0052 itself. The existing language in ADR-0052 already treats it as directional, not binding. ADR-0080 formalizes this distinction.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The ADR correctly takes a non-breaking approach: existing ADRs are not retroactively non-compliant. Future amendments will decompose infra-specific guidance. This is pragmatic and avoids a cascade of immediate amendments.

### Assumption 3.3: Infrastructure ADRs can use the same governance process

- **Stated Norm:** "Infrastructure ADRs govern components deployed and operated independently of the FastAPI process. They follow the same ADR-0044 governance process."
- **Underlying Assumption:** ADR-0044's governance model (tiers, metadata, challenge review) works equally well for infrastructure decisions as for application decisions.
- **Challenge:** Infrastructure decisions may have different stakeholders (DevOps, platform team), different change cadence (Terraform changes vs. app code changes), and different review criteria (cost, availability, blast radius). Does the ADR-0044 process fit?
- **Evidence Strength:** ⭐⭐ Moderate
- **Counter-Evidence Found:** Partial — ADR-0044's tier system was designed for application architecture tiers (Tier-1 principles → Tier-5 migrations). Infrastructure decisions don't clearly map to "Domain Contract" (Tier-3) or "Feature Decision" (Tier-4). However, ADR-0044's metadata schema and review process are content-neutral. The tier definitions may need future refinement for infrastructure ADRs, but this is a follow-up concern, not a blocker.
- **Confidence (ADR survives challenge):** 🟡 Moderate
- **Reviewer Notes:** This is a valid concern but not a blocker for ADR-0080. The boundary principle is correct regardless of whether infrastructure ADRs need their own tier definitions. The ADR explicitly states it "does not create infrastructure-specific ADRs" — it enables them. If the tier system needs adaptation for infra ADRs, that can be addressed when the first infra ADR is authored.

### Assumption 3.4: The portability invariant is achievable

- **Stated Norm:** "The application ADR corpus must remain valid and enforceable if the hosting platform changes entirely."
- **Underlying Assumption:** The application ADR corpus does not contain any norms that are inherently tied to a specific hosting platform.
- **Challenge:** ADR-0045 P1 references "ECS tasks" in its principle text. ADR-0052 references ECS throughout. Are there application ADRs so deeply tied to AWS/ECS that the portability invariant would retroactively invalidate them?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No — ADR-0045 P1 mentions "ECS tasks" as context for the stateless process principle, not as a normative requirement. The principle itself ("stateless and share-nothing") is platform-agnostic. ADR-0052's ECS references are already classified as implementation guidance, not as the standard itself (the standard is "strictly separate build, release, and run stages"). The portability invariant would correctly flag these as infrastructure-domain context, not as violations requiring ADR rewrites.
- **Confidence (ADR survives challenge):** 🟢 High

## 7. Tradeoffs Accepted

### Tradeoff 7.1: Governance overhead vs. scope clarity

- **Chosen:** Require domain classification (app vs. infra) before authoring any new ADR.
- **Rejected:** No classification requirement; infer scope from content.
- **Rationale:** Explicit classification prevents the scope ambiguity that motivated this ADR. The overhead is minimal (one classification step) compared to the cost of retroactive scope corrections.
- **Risk Accepted:** Authors may initially struggle with classification for straddling concerns.
- **Contingency:** Principle 2's contract table pattern provides decomposition guidance. Challenge review catches misclassification.

### Tradeoff 7.2: Non-breaking adoption vs. immediate compliance

- **Chosen:** Reclassify existing infra-specific guidance as "informational context" without immediate amendment.
- **Rejected:** Require immediate amendment of all affected ADRs (ADR-0052, ADR-0054) to extract infra-specific guidance.
- **Rationale:** Immediate amendment would block the boundary principle's adoption on a cascade of lower-priority edits. The informational reclassification provides the governance authority without the editorial burden.
- **Risk Accepted:** Developers may still treat informational infra-specific guidance as normative until amendments occur.
- **Contingency:** Follow-up actions list ADR-0052 as a candidate for decomposition. Challenge review of future work citing infra-specific guidance will apply the boundary principle.
