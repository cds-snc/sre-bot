## 3. Assumptions Challenged

### Assumption 3.1: Cross-Package Import Prohibition Is Sufficient Without a Communication Substitute

- **Stated Norm:** "Feature packages must not import from sibling feature packages. The only permitted dependency direction from a feature package is downward into `app/infrastructure/`." (Standard 2)
- **Underlying Assumption:** Infrastructure-mediated communication (events, storage, hookspec orchestration) is always a viable substitute for direct cross-package imports, and the overhead of that mediation is acceptable for all use cases.
- **Challenge:** For purely read-only domain-type references (e.g., Package B needs to display a label derived from Package A's enumeration), forcing infrastructure mediation adds real coupling to infrastructure with no behavioral benefit. A shared event bus is appropriate for async coordination; it is heavy machinery for resolving a label string. The assumption that mediation is always appropriate conflates coordination patterns with type-sharing needs.
- **Evidence Strength:** ⭐⭐ Moderate — PEP 484 and Python typing design explicitly support `TYPE_CHECKING`-only imports for exactly this scenario (type-only cross-module references without runtime dependency). Bogard's vertical slice architecture, however, supports prohibition at slice boundaries as a forcing function toward explicit contracts.
- **Counter-Evidence Found:** Yes — PEP 484 Section "Circular Imports" specifically endorses `TYPE_CHECKING` guards as the idiomatic Python solution when types must be referenced across modules that cannot have runtime circular dependencies. The ADR's S2.2 prohibition extends beyond runtime coupling to type-system coupling, which is more controversial.
- **Confidence (ADR survives challenge):** 🟢 High — The prohibition is defensible as an architectural forcing function. The codebase has a precedent (`app/modules/`) demonstrating what happens without it. However, the rationale for S2.2 specifically should be stated more explicitly in the ADR: the issue is *semantic coupling* (type definitions become an implicit contract between packages) not runtime behavior. The current ADR simply states the prohibition without explaining why `TYPE_CHECKING` imports are also prohibited.
- **Reviewer Notes:** The prohibition is correct. The ADR's stated rationale ("packages must communicate through infrastructure-mediated mechanisms") is the right answer but doesn't fully explain *why* a `TYPE_CHECKING` import specifically violates slice isolation. Recommend adding one sentence: "Even without runtime coupling, `TYPE_CHECKING` imports bind the type-checker to another package's internal model, creating an implicit interface contract that breaks when the referenced type changes."

### Assumption 3.2: Shared Kernel Restricted to Value Types Is Sufficient for Intra-Bounded-Context Sub-Packages

- **Stated Norm:** "The shared kernel may contain only: value types (dataclasses, enums), event definitions, settings classes, and provider functions. It must never contain service implementations." (S3.4) — AND — Standard 4's "Allowed" column includes "Cross-subfeature Protocol contracts."
- **Underlying Assumption:** Value types, events, settings, providers, and Protocol contracts (from Standard 4) are the correct and complete set of shareable artifacts for an intra-bounded-context shared kernel.
- **Challenge:** **Internal inconsistency detected.** S3.4 enumerates the allowed content of `common/` and explicitly excludes service implementations. Standard 4's "What belongs in the internal shared layer" table adds "Cross-subfeature Protocol contracts" to the allowed list. These two rules govern the same artifact (`common/`) but S3.4's enumeration does not include Protocol contracts. The codebase confirms this issue: `packages/access/common/config/loaders.py` contains `AccessConfigLoader`, a `Protocol` class, which Standard 4 implicitly permits but S3.4's enumeration does not. A developer reading S3.4 in isolation would conclude Protocols are not allowed in `common/`.
- **Evidence Strength:** ⭐ Strong — The inconsistency is textually verifiable by comparing S3.4's enumeration with Standard 4's table, and confirmed by the actual `access/common/config/loaders.py` file in the codebase.
- **Counter-Evidence Found:** Yes — Standard 4 explicitly permits cross-subfeature Protocol contracts in `common/`, which Standard 3 does not. ADR-0065 Principle 2 confirms Protocol types are the correct mechanism for cross-component contracts, supporting the Standard 4 interpretation.
- **Confidence (ADR survives challenge):** 🟡 Moderate — The *intent* is clear from Standard 4 and confirmed by the codebase. Standard 4 supersedes S3.4 for this specific case because it is more specific. However, the inconsistency creates real ambiguity for implementers. **This is the primary revision action required** (non-blocking for gate, but must be resolved before Step 10 cascade).
- **Reviewer Notes:** S3.4 must be updated to add "cross-subfeature Protocol contracts" to the allowed list, or reference Standard 4 for the complete specification. Evans' Shared Kernel concept also supports including Protocol contracts in the kernel, as they represent the "agreed-upon interface" between bounded context sub-domains.

### Assumption 3.3: Capability Promotion Requiring All Three Triggers Is Appropriately Conservative

- **Stated Norm:** "Promotion triggers (all three must be satisfied): T1: Multi-consumer demand (≥2 independent feature packages), T2: Operational centralization (policy/SLA/audit/security requires central ownership), T3: Contract stabilization (interface does not reference feature-specific types)." (Standard 6)
- **Underlying Assumption:** Premature extraction is always worse than temporary duplication, even when security or compliance mandates central ownership (T2) before a second consumer exists (T1).
- **Challenge:** The AND requirement across all three triggers creates a governance gap for security-critical capabilities. If a security policy mandates that AWS Identity Store access must be centrally owned and audited (T2 is strongly satisfied), but only one feature currently needs it (T1 not yet satisfied), the ADR requires the capability to remain feature-local until a second consumer appears. This could create an unbounded window where a security-mandated central ownership policy cannot be enforced because T1 is not yet met. The Provisioning feature example in the ADR perfectly illustrates this: if AWS mandates that all Identity Store API calls route through a centrally audited adapter for compliance, the ADR's rules block extraction until another feature also needs the capability.
- **Evidence Strength:** ⭐⭐ Moderate — Fowler's YAGNI and "Sacrifice One" principle support requiring multi-consumer demand. However, Fowler's principle is about *premature abstraction*, not *compliance-driven extraction*. Security/compliance extraction is a different driver than convenience abstraction.
- **Counter-Evidence Found:** Yes — ADR-0064 (Security) establishes defense-in-depth as a governing principle. Infrastructure-layer centralization of security-critical API access is a defense-in-depth mechanism. The three-trigger rule could conflict with ADR-0064 by blocking centralization of a security-critical capability until a second feature consumer emerges.
- **Confidence (ADR survives challenge):** 🟡 Moderate — The three-trigger rule is correct for the primary case (convenience/DRY-driven extraction). However, a T2-override path should be acknowledged: when T2's "policy, SLA, audit, or security requirements mandate central ownership" is the dominant driver, T1 should not be required. The current framing forces all three, with no exception for mandatory governance extraction.
- **Reviewer Notes:** Recommend adding a parenthetical to Standard 6: "Exception: when T2 is driven by mandatory security, audit, or compliance policy that requires central ownership regardless of consumer count, T1 may be waived with explicit documentation of the security driver in the promotion ADR." This is a clarification, not a structural change.

### Assumption 3.4: TYPE_CHECKING Imports From Sibling Packages Are Prohibited Without Exception

- **Stated Norm:** "`TYPE_CHECKING` imports from sibling packages are prohibited. If a package needs a type from another package's domain, the packages must communicate through infrastructure-mediated mechanisms." (S2.2)
- **Underlying Assumption:** Any reference to another package's types, even type-checker-only references, constitutes package coupling that infrastructure-mediation can always replace.
- **Challenge:** Infrastructure-mediated communication (events, hookspecs, storage) is behavioral coordination, not type description. When Package B's service method must return a value that an infrastructure service expects to carry a type from Package A (e.g., a generic event payload typed with Package A's domain type), S2.2 prohibits the type annotation while requiring that the same semantic dependency exist at runtime through the event's data structure. This creates a situation where the runtime coupling exists (via the event payload's shape) but the type system cannot express it — decreasing rather than increasing safety.
- **Evidence Strength:** ⭐⭐ Moderate — Python typing design explicitly supports `TYPE_CHECKING` as a zero-runtime-overhead way to express such dependencies. However, Bogard's vertical slice principle supports the prohibition: if Package B genuinely needs to reference Package A's type, the two packages are in the same bounded context and should share a `common/` kernel, or the type belongs in infrastructure.
- **Counter-Evidence Found:** Yes — Mypy documentation explicitly shows `TYPE_CHECKING` imports as the solution for "type-only circular dependencies." The PEP 484 design explicitly anticipates cross-module type references without runtime coupling.
- **Confidence (ADR survives challenge):** 🟢 High — The prohibition survives challenge because the ADR correctly identifies the resolution path: if two packages genuinely need to share a type, that type belongs either in infrastructure (if domain-neutral) or in a `common/` kernel (if intra-bounded-context). The prohibition is a forcing function that prevents the gradual accumulation of implicit inter-package type dependencies. The ADR's own alternatives section correctly rejects ad-hoc shared modules by citing the `app/modules/` failure mode.
- **Reviewer Notes:** The ADR would benefit from acknowledging that `TYPE_CHECKING` imports *feel* harmless, then explaining the architectural reason they are prohibited: they bind the type system of Package B to the internal model of Package A, which undermines independent evolution and is semantically equivalent to a runtime import from an architectural design standpoint.

## 4. Failure Modes Identified

### Failure Mode 4.1: S3.4/Standard 4 Inconsistency Causes Protocol Types to Be Rejected From `common/`

- **If Assumption Fails:** A developer reads S3.4's enumeration (value types, event definitions, settings, provider functions) and correctly concludes that `Protocol` classes are not in the list. The developer moves `AccessConfigLoader` or an equivalent Protocol out of `common/` and into a sub-package, breaking the intra-bounded-context contract.
- **Platform Impact:**
  - Incident management workflow: None
  - Access synchronization workflow: Medium — if `AccessConfigLoader` protocol is moved into `sync/`, it can no longer be consumed by `request/` or `catalog/`, forcing duplication or a structural violation
  - Access request workflow: Medium — same as sync
  - Multi-provider integrations: None
- **Probability Estimate:** Medium — the inconsistency is detectable by reading the ADR carefully, and Protocol is a natural thing to put in a shared kernel
- **Mitigation or Acceptance:** **Mitigation required** — Fix S3.4 to include "cross-subfeature Protocol contracts" in the allowed list. This aligns S3.4 with Standard 4 and the existing codebase.

---

### Failure Mode 4.2: T2-Only Security Promotion Blocked by T1 Requirement

- **If Assumption Fails:** A security audit requires centralized ownership of an infrastructure capability currently living in Feature A. Feature B does not exist yet, so T1 is unsatisfied. The ADR's three-trigger requirement blocks the compliance-driven extraction, leaving the capability feature-local against security policy.
- **Platform Impact:**
  - Incident management workflow: Low
  - Access synchronization workflow: High — AWS Identity Store reconciliation is the specific example; if compliance mandates centralization before Feature B arrives, there is no ADR-compliant path
  - Access request workflow: Low
  - Multi-provider integrations: Medium — OAuth token management or credential rotation could fall into this gap
- **Probability Estimate:** Low-Medium — depends on regulatory/compliance environment; if AWS governs API access policies centrally, this becomes a real constraint
- **Mitigation or Acceptance:** Accepted as documented risk. Recommend adding a T2-override exception clause (see Section 3, Assumption 3.3 Reviewer Notes). If the risk materializes, a waiver note in the promotion ADR is the escape hatch.

---

### Failure Mode 4.3: TYPE_CHECKING Prohibition Creates Undocumented Event Payload Typing Gap

- **If Assumption Fails:** When Package B listens to an event published by Package A, the event payload carries data whose type is Package A's domain type. Package B's handler must work with an untyped or loosely typed payload (e.g., `Any` or `dict`), losing type safety at the cross-package boundary. Infrastructure-mediated events require the event schema to be defined in infrastructure, meaning Package A's domain type must be either duplicated or generalized.
- **Platform Impact:**
  - All workflows: Low to Medium — type safety degradation at event boundaries increases risk of runtime errors when event schemas evolve
- **Probability Estimate:** Medium — as the event system grows, this pattern is likely to appear
- **Mitigation or Acceptance:** Accepted as intentional tradeoff. The resolution is to define event schemas as infrastructure-owned `@dataclass(frozen=True)` types (ADR-0065 P3), not package-domain types. The ADR could benefit from a one-sentence note stating: "Cross-package event schemas must be defined as infrastructure-owned domain-neutral types, not as references to feature-domain types."

## 7. Tradeoffs Accepted

### Tradeoff 7.1: Isolation Purity vs. Developer Ergonomics (TYPE_CHECKING Prohibition)

- **Chosen:** Prohibit `TYPE_CHECKING` imports between packages (S2.2); all cross-package type needs resolved via infrastructure-owned types or `common/` shared kernels
- **Rejected:** Permit `TYPE_CHECKING`-only imports between packages (zero runtime cost, type-safe)
- **Rationale:** `TYPE_CHECKING` imports, though runtime-free, bind the type system of Package B to the internal model of Package A. This prevents independent evolution of each package's domain model and creates an implicit interface contract. The `app/modules/` failure mode demonstrates the cost of incremental "harmless" cross-imports over time.
- **Risk Accepted:** Developers may find it frustrating to route simple type references through infrastructure when the types feel local. Event payload typing at cross-package boundaries becomes less precise.
- **Contingency:** When two packages need to share a type (not just reference it), the type belongs in infrastructure or in a `common/` kernel if intra-bounded-context. The escape hatch is standard promotion lifecycle, not an import exception.

---

### Tradeoff 7.2: Friction-Heavy Promotion vs. Premature Extraction (Three-Trigger Rule)

- **Chosen:** Require all three triggers (multi-consumer demand, operational centralization, contract stabilization) before capability promotion
- **Rejected:** Promote ad hoc when need arises; or a single-trigger model (e.g., T2 alone)
- **Rationale:** Premature extraction creates infrastructure services with no stable contract, owned by a team with no clear operational mandate. Fowler's "Sacrifice One" principle supports temporary duplication over wrong abstraction. The three-trigger rule ensures promotion is driven by real, recurring need.
- **Risk Accepted:** A small window exists where T2 alone (security/compliance mandate) warrants centralization before a second consumer appears. This creates a governance gap documented in Failure Mode 4.2.
- **Contingency:** The promotion ADR (S6.2) can document a T2-override waiver with explicit security justification. The three-trigger rule is a default, not an absolute constraint in the face of mandatory governance requirements.

---

### Tradeoff 7.3: Shared Kernel Constraint (Value Types + Protocols Only) vs. DDD Full Shared Kernel

- **Chosen:** Restrict `common/` to value types, events, settings, provider functions, and cross-subfeature Protocol contracts
- **Rejected:** Allow any shared domain element in `common/` (Evans' full Shared Kernel pattern)
- **Rationale:** In a non-distributable monolith where sub-packages are internal decomposition within one bounded context, the risk of `common/` becoming a "second service layer" is real. Restricting to stable artifacts prevents gradual entanglement.
- **Risk Accepted:** Occasionally, a business logic function is truly shared across sub-packages and must be duplicated or refactored into sub-package ownership. The "Negative" consequence section acknowledges this.
- **Contingency:** S4.2 provides the refactoring path: if a shared concern evolves into service logic, move it to the sub-package that owns the behavior.

---

### Tradeoff 7.4: Top-Level Test Tree vs. Colocated Tests

- **Chosen:** All tests in `app/tests/` tree mirroring package paths (Standard 5)
- **Rejected:** Colocated tests inside each package directory
- **Rationale:** Packages are not independently distributable. Fixture sharing requires cross-package test imports when tests are colocated. Pytest "Good Integration Practices" recommends the external tree for applications.
- **Risk Accepted:** Test location is more distant from source. Developers must remember the parallel directory structure.
- **Contingency:** The directory mirroring convention (`unit/packages/X/` mirrors `packages/X/`) minimizes cognitive overhead.
