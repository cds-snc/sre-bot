## 3. Assumptions Challenged — Standards 7–8

### Assumption 3.1: The Realization / Transport Split Is Consistently Applicable in Practice

- **Stated Norm:** "A class that mixes Realization and Transport responsibilities must be split." (S7.1)
- **Underlying Assumption:** The distinction between planning/diffing/sequencing (Realization) and API calls/pagination/retry (Transport) is recognizable and consistently applicable at code review time.
- **Challenge:** In practice, the Realization / Transport boundary is fuzzy for some cases. E.g., a Reconciler that calls `ClientAdapter.get_user()` to check existence before deciding to call `ClientAdapter.create_user()` — is the existence check a Realization decision (entity matching) or a Transport concern (fetching remote state)? The get-then-decide pattern mixes the two layers within a single Reconciler method.
- **Evidence Strength:** ⭐ Strong — `AwsIdentityCenterAdapter` demonstrates exactly this pattern: it fetches group membership to compute a diff, which spans both layers.
- **Counter-Evidence Found:** Yes — S7.4 explicitly acknowledges the Reconciler may contain decision logic, which means decision logic that requires remote state is *legitimately* in the Reconciler layer. The get-before-decide pattern does not violate S7 because the Reconciler calls the Client Adapter (downward dependency allowed). S7.2 is satisfied: only the ClientAdapter imports SDK types; the Reconciler calls the ClientAdapter's domain-typed API.
- **Confidence (ADR survives challenge):** 🟢 High — the layer model is operationally clear: if a class makes direct SDK calls AND contains planning logic, it must be split. A class that exclusively makes SDK calls via an injected ClientAdapter and contains planning logic is correctly classified as a Reconciler. S7.1–S7.4 together define this boundary unambiguously.
- **Reviewer Notes:** No revision needed. S7.4 is the key constraint that makes the distinction actionable.

---

### Assumption 3.2: S7.5 Exemption ("Thin Outbound") Is Self-Policing

- **Stated Norm:** "For features with only thin API translation and no reconciliation logic, the Reconciler layer may be omitted." (S7.5)
- **Underlying Assumption:** The distinction between "thin API translation" and "reconciliation logic" is sufficiently clear that developers self-classify correctly without needing a decision checklist.
- **Challenge:** A notification dispatch function that formats a Slack message and calls the Slack API *appears* thin, but could accumulate logic over time (conditional formatting, fallback channels, retry with different payloads) that crosses into Reconciler territory. Developers invoking S7.5 may not revisit the classification as the class grows.
- **Evidence Strength:** ⭐⭐ Moderate — class growth is a common accumulation pattern; S7.5 exemptions become technical debt if not re-evaluated.
- **Counter-Evidence Found:** Yes — S8.1 provides the re-evaluation trigger: if a class named `*Adapter` accumulates reconciliation logic (entity matching, diffing, sequencing), it becomes a code review red flag. The named-role enforcement (Standard 8) is the policing mechanism that catches S7.5 exemptions that have outgrown their classification.
- **Confidence (ADR survives challenge):** 🟢 High — S7.5 and S8.1 are complementary: S7.5 grants the exemption; S8.1 enforces the boundary. The pairing is sound.
- **Reviewer Notes:** No revision needed. A note in S7.5 cross-referencing S8.1 would be a helpful editorial addition but is not normatively required.

---

### Assumption 3.3: `<Platform>Reconciler` Naming Is Sufficient to Catch Drift at Code Review

- **Stated Norm:** "A class named `*Adapter` that contains reconciliation logic (entity matching, diffing, sequencing) is a code review red flag and must be renamed or split." (S8.1) "Protocol contracts for reconcilers and client adapters must use role-specific names." (S8.3)
- **Underlying Assumption:** Code reviewers can reliably identify whether a class contains "reconciliation logic" from a PR diff — without needing to audit the entire class history.
- **Challenge:** For incremental PRs, a class may receive a small reconciliation method in one PR that does not trigger a rename (the class is still mostly thin). The "red flag" condition accumulates gradually.
- **Evidence Strength:** ⭐⭐ Moderate — incremental accumulation is a real pattern in long-lived codebases.
- **Counter-Evidence Found:** Yes — S8.1's test criterion ("contains reconciliation logic: entity matching, diffing, sequencing") is observable from the class body, not the PR diff alone. Code review can inspect the full class in context. The standard does not say "only triggered by new methods" — it applies to the class as it exists at review time.
- **Confidence (ADR survives challenge):** 🟢 High — the naming enforcement rule is correctly scoped to the class at review time, not just to the incremental change. Reviewers who apply S8.1 holistically will catch accumulation. No revision needed.

## 4. Failure Modes Identified

### Failure Mode 4.1: `AwsIdentityCenterAdapter` Split Indefinitely Deferred

- **If Assumption Fails:** The `AwsIdentityCenterAdapter` migration (Implementation Guidance #3) is never executed, leaving a mixed-responsibility class in production while ADR-0088 S7.1 marks it as non-compliant.
- **Platform Impact:** Access Sync workflow: Medium — the mixed class creates architectural drift but does not break functionality. It creates hidden decision logic that is harder to test and extend.
- **Probability Estimate:** Medium — migration items without hard deadlines are frequently deferred.
- **Mitigation:** Implementation Guidance #3 is clear and actionable. A work item should be tracked in the project backlog. No change to the ADR is required; governance exists.

---

### Failure Mode 4.2: S7 Applied Too Broadly (Over-Engineering Thin Integrations)

- **If Assumption Fails:** Developers apply S7 (three-layer model) to trivially thin outbound integrations (e.g., a single `requests.get()` call to fetch geo-data), creating unnecessary Reconciler + ClientAdapter pairs.
- **Platform Impact:** New feature development velocity: Low — the S7.5 exemption explicitly prevents this.
- **Probability Estimate:** Low — S7.5 is clearly stated and the scenario table provides concrete guidance.
- **Mitigation:** S7.5 + scenario table is the mitigation. No change needed.
