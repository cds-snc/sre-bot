## 2. Core Finding — Category A Classification Failure

**ADR-0077 Category A criteria:**

1. The service abstracts over an external backing service.
2. Feature packages depend on the service and would need to change if the backing service changed.

**`IdentityService` against these criteria:**

| Criterion | Assessment |
|-----------|-----------|
| Abstracts over external backing service? | **No.** `resolve_from_jwt` performs 6 dict lookups on an already-validated, in-memory payload. Zero network calls. The JWKS endpoint is accessed by `JWKSManager` upstream, not by `IdentityService`. |
| Feature packages depend on it? | **No.** Zero imports of `IdentityServiceDep` found in `app/packages/**` or `app/api/**`. The only production caller is a single line in `security/current_user.py` — an infrastructure-internal call, not a feature-package boundary crossing. |

**Conclusion:** `IdentityService` never satisfied Category A criteria. Its classification as Category A in the original ADR-0077 table was an error inherited from ADR-0023 framing ("identity = service") without applying the criteria. The correct disposition is removal from the Category A table entirely.

## 3. Assumptions Challenged

### Assumption 1: Removing `IdentityService` from the DI container does not regress testability in feature packages

**Claim:** All feature-package tests already override `get_current_user` directly via `app.dependency_overrides`. None of them mock `IdentityService`. Removing it has no effect on test coverage.

**Verification:** Grepped `app/packages/**` and `app/api/**` for `IdentityServiceDep` — zero results. Confirmed the three active feature test files (`test_access_request_routes.py`, `test_access_sync_routes.py`, `test_access_catalog_routes.py`) all use `app.dependency_overrides[get_current_user]`.

**Confidence:** 🟢 High

---

### Assumption 2: `_build_user_from_jwt_payload` is correctly classified as a private module function, not a service

**Claim:** PEP 544 Protocol contracts add value only when two conditions hold: (a) the operation has a real alternative implementation, and (b) the caller needs to be decoupled from the specific implementation. `_build_user_from_jwt_payload` fails both. It is a pure transform of a validated, in-memory dict. There is no alternative implementation because there is nothing to substitute.

**Standard reference:** ADR-0077 Category A Criterion 1: "The service has at least one plausible alternative implementation (even if not currently built)." A function that reads `payload["sub"]` has no alternative implementation concept.

**Confidence:** 🟢 High

---

### Assumption 3: `User` and `AuthPrincipalSource` belong in `infrastructure/security/models.py`

**Claim:** `User` is the authenticated HTTP principal — the return type of `get_current_user()`. It is produced by the security layer and consumed at the HTTP transport boundary. Locating it in `infrastructure/security/models.py` reflects true ownership. `AuthPrincipalSource` is the enum on `User.source`; it travels with `User`. The rename from `IdentitySource` to `AuthPrincipalSource` better describes what the enum means: the authentication transport mechanism that vouched for the current request.

**Import path change impact:** 17 import statements across 10 files, all confirmed in greenfield code (`packages/`, `infrastructure/security/`, test files). Zero imports in frozen zones (`modules/`, `integrations/`) confirmed by grep.

**Confidence:** 🟢 High

---

### Assumption 4: Dissolving `IdentityResolver` class does not leave future transport owners without a pattern

**Claim:** `resolve_from_slack`, `resolve_from_webhook`, and `resolve_system_identity` have zero production callers in the greenfield codebase. When Phase 3 implements `SlackPlatformProvider.resolve_user()`, it will write the Slack user lookup directly in the platform provider — not by resurrecting `IdentityResolver`. The ADR-0067 S8 standard documents what that function must do.

**Confidence:** 🟢 High

---

### Assumption 5: Removing `IdentityService` from ADR-0077 Standard 1 does not affect other Category A services

**Claim:** The `IdentityService` row in Standard 1 has no downstream dependencies. No other row references it. Its removal is self-contained.

**Verification:** Read all Category A rows — none reference `IdentityService`. Read Standard 5 migration table — `IdentityService` row is also standalone.

**Confidence:** 🟢 High

---

### Assumption 6: ADR-0067 S8 (Slack user identity resolution) is independent and unaffected

**Claim:** ADR-0067 S8 governs `SlackPlatformProvider.resolve_user()` — a future Phase 3 implementation. It never referenced `IdentityService` or `IdentityResolver` as its mechanism. It uses `AuthPrincipalSource.SLACK` (the enum value), which survives the rename from `IdentitySource.SLACK`.

**Confidence:** 🟢 High

## 5. ADR-0064 Follow-up (Non-Blocking)

Standard 10 of ADR-0064 references "governed by ADR-0061 Standard 3 and ADR-0078" for platform-transport authentication. After this amendment, ADR-0061 Standard 3 governs JWT claim extraction. The platform-transport reference should be updated to "governed by ADR-0067 Standard 8 and ADR-0078." This is an editorial amendment to ADR-0064; it does not affect the security boundary.
