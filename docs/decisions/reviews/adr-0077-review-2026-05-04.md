## 3. Assumptions Challenged

### Assumption 3.1: Features should choose based on domain coverage, not service tier

- **Stated Norm:** "Features choose based on domain coverage, not service tier. If a Category A service covers the operation, use it (S3.1). If the operation is domain-specific and no Category A service exists, use the client facade directly through the injection boundary (S3.2)."
- **Underlying Assumption:** The deciding factor for whether to use a core service vs a client facade is whether an abstraction *already exists* that covers the needed operation, not the tier classification of the dependency.
- **Challenge:** Could this lead developers to bypass Category A services when they *should* be using them, by claiming the operation is "domain-specific"?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No. S3.4 explicitly states "If a Category A service covers the operation, use it." The "client facade" path is only available when "no domain abstraction exists." Additionally, the existing ADR-0077 text includes a "two consumers" trigger: if two features need the same domain-specific operation, that triggers creating a new Category A service. This prevents drift.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The domain-coverage criterion is testable during code review: does a Category A Protocol already have this method? If yes, use it. If no, client facade is permitted.

### Assumption 3.2: Client facades are architectural acceptable as direct dependencies

- **Stated Norm:** "Client facades are pre-authenticated SDK wrappers. They own connection credentials and session management."
- **Underlying Assumption:** Depending directly on a client facade (e.g., `AWSClients.identitystore`) from a feature package is architecturally sound because the client facade already handles credential management, and the feature only uses it for domain-specific operations.
- **Challenge:** Does direct client facade access violate the hexagonal architecture principle that application core should not depend on adapters?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** Partial. Hexagonal architecture purists would argue every external access should go through a port. However, S3.4 explicitly documents this as a "pragmatic exception" with known tradeoff: the feature has a concrete dependency on the facade. The ADR's existing S3.2 already governs this exception pattern with documentation requirements. Cosmic Python also takes the pragmatic approach — not every external call gets a repository abstraction.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The pragmatic exception is well-documented and bounded. It applies only when no Category A service exists. The "two consumers" trigger provides a natural escalation path to abstraction when needed.

### Assumption 3.3: The distinction between client facades and core services is clear

- **Stated Norm:** "Client facades are pre-authenticated SDK wrappers owning credentials. Core services are domain abstractions composed from pre-configured clients."
- **Underlying Assumption:** The architectural distinction between these two concepts is unambiguous enough to guide developer decisions.
- **Challenge:** Could there be borderline cases where a service is both a credential owner and a domain abstraction?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No. The codebase evidence is clear: `AWSClients` owns boto3 sessions and credentials (client facade). `DirectoryProvider` receives pre-configured `GoogleWorkspaceClients` via constructor injection and adds domain logic (core service). The distinction maps to a concrete structural difference: does the constructor accept settings/credentials directly (facade), or does it accept already-constructed clients (core service)?
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The structural test (accepts settings/credentials vs accepts pre-built clients) is a reliable discriminator.

### Assumption 3.4: Cross-reference to ADR-0056 S3 is correct

- **Stated Norm:** "For the corresponding provider location rules — where providers are defined vs where they are consumed — see ADR-0056 Standard 3."
- **Underlying Assumption:** ADR-0056 S3 and ADR-0077 S3.4 are complementary: S3.4 governs *what* features consume; ADR-0056 S3 governs *where* providers are defined and *how* they are consumed (import path).
- **Challenge:** Could the cross-reference create circular governance?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No. The concerns are clearly separated: ADR-0077 classifies services and defines *which* pattern to use; ADR-0056 defines *where provider functions live* and *how features import them*. No circularity — they govern orthogonal decisions.
- **Confidence (ADR survives challenge):** 🟢 High

## 7. Tradeoffs Accepted

### Tradeoff 7.1: Pragmatism vs Hexagonal Purity

- **Chosen:** Permit direct client facade access for domain-specific operations (S3.4 + S3.2).
- **Rejected:** Require a Category A Protocol for every external operation.
- **Rationale:** Creating thin wrapper Protocols for every AWS/Google/Slack API call would produce hundreds of single-method Protocols that add indirection without substitution value. Only operations with genuine domain abstraction benefit from a Protocol.
- **Risk Accepted:** Features using client facades have concrete dependencies that cannot be swapped without code changes. This is acceptable because these are domain-specific operations that are unlikely to change backing implementations.
- **Contingency:** The "two consumers" trigger escalates to a Category A service when the abstraction would serve multiple features.
