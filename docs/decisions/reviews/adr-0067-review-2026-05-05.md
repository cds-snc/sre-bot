## 3. Assumptions Challenged

### Assumption 1: Slack `user_id` → `User` resolution has no owner before this amendment

**Claim:** Before S8, the Slack user resolution responsibility was assigned to `IdentityService` in ADR-0061. After the ADR-0061 S3 amendment (Protocol narrowed to JWT-only), this responsibility had no governed owner. S8 fills the gap.

**Challenge Method:** Search ADR corpus for any standard that governed Slack `user_id` → `User` resolution before this amendment.

**Result:** ADR-0061 (pre-amendment) Standard 2 listed "Platform" as one of four resolution sources, but assigned it to `IdentityService` (the old orchestrator). ADR-0067 had no prior standard for user resolution. ADR-0078 governs platform providers generically but does not mention user identity resolution specifically. S8 is genuinely new governance, not duplication.

**Confidence:** 🟢 High

---

### Assumption 2: Email is the correct canonical key for Slack → `User` resolution

**Claim:** The canonical `User` model uses `email` as the primary key. Slack `users.info` returns an `email` field (requires `users:read.email` scope). Email is the correct join key from Slack identity to canonical identity.

**Challenge Method:** Review `app/infrastructure/identity/models.py` `User` model fields. Verify email is present. Review Slack `users.info` response schema.

**Result:** `User` model has `email: str` as a primary identifying field (used for IDP lookups in `DirectoryProvider`). Slack `users.info` returns `profile.email` when the bot has `users:read.email` scope. Email is the correct and available join key. Absent email (bot users, certain guest accounts) must fail gracefully per S8 R4.

**Confidence:** 🟢 High

---

### Assumption 3: Category C classification means S8 does not trigger a Protocol migration obligation

**Claim:** S8 classifies `SlackPlatformProvider` user resolution as Category C. This means: no Protocol is required, no `IdentityServiceProtocol` extension is needed, and the capability does not appear in the ADR-0077 Standard 1 migration priority table as a P1 item.

**Challenge Method:** Review ADR-0077 Standard 1 Classification Criteria for Category C.

**Result:** ADR-0077 Standard 1 states Category C is "Implementation Detail — never exposed to features, no Protocol required, no migration priority." S8 R1 is consistent. Feature code must never import `SlackPlatformProvider` directly for user resolution — it receives the resolved `User` from the Slack handler context, not by calling the provider.

**Confidence:** 🟢 High

---

### Assumption 4: No-cache-across-requests rule (S8 R5) is operationally sound

**Claim:** Slack user profiles can change (name, email, deactivation). Per-request resolution without a shared cache is the safest posture. Per-request context caching is acceptable for performance within a single Slack event handling cycle.

**Challenge Method:** Assess whether Slack profile change frequency justifies no shared cache.

**Result:** Slack `users.info` response does not include a cache control header. User profiles change infrequently in practice, but deactivation can happen at any time and should be reflected immediately. Given that Slack commands are low-frequency (human-triggered), the performance cost of one `users.info` call per command is negligible. No shared TTL cache is the correct tradeoff.

**Confidence:** 🟢 High

---

### Assumption 5: S8 does not conflict with existing S6 (Outbound Message Delivery)

**Claim:** S6 governs outbound message delivery (provider exposes `client` for `chat_postMessage`). S8 governs inbound identity resolution (`user_id` → `User`). These are orthogonal concerns on the same provider.

**Challenge Method:** Read S6 rules for any statement about identity or user resolution.

**Result:** S6 R1–R4 are entirely about outbound messaging (`client`, `chat_postMessage`, `SlackApiError` handling, legacy migration). No identity or `users.info` content. S8 is entirely about inbound identity. No overlap.

**Confidence:** 🟢 High

---

### Assumption 6: The `user_id` in Socket Mode events is trusted without per-message signing

**Claim:** The application uses Socket Mode (WebSocket transport, `SLACK_APP_TOKEN`). Individual event messages delivered over the WebSocket are NOT HMAC-signed — unlike HTTP Events API mode where each POST has `X-Slack-Signature`. The `user_id` in a Bolt event payload is trusted because the WebSocket connection itself was authenticated by Slack.

**Challenge Method:** Review Slack Socket Mode documentation and ADR-0067 S5-R2 and S5-R5. Confirm that `SLACK_SIGNING_SECRET` is not used for per-message verification in Socket Mode.

**Result:**
- ADR-0067 S5-R2 explicitly states `SLACK_SIGNING_SECRET` is optional, "only needed if HTTP webhook verification is added in future."
- ADR-0067 S5-R5 states `SLACK_SOCKET_MODE` defaults to `True`. When `False`, no `SocketModeHandler` is created.
- Slack documentation confirms: in Socket Mode, the app authenticates once with `SLACK_APP_TOKEN` to establish the WebSocket tunnel. Slack signs the initial handshake; individual events are not signed. Trust is by virtue of the authenticated channel.
- S8 R1 correctly requires no per-message verification assumption. The `user_id` is provisioned by Slack and arrives over the authenticated channel.
- The original draft text "confirmed by Slack request signing" was inaccurate and has been corrected in S8 to reflect the Socket Mode trust model.

**Implication for ADR-0061 Standard 2:** The table cell previously read "Slack `user_id` from signed request" — this was HTTP Events API terminology and is incorrect for Socket Mode. Corrected to "Slack `user_id` from Bolt event payload (trusted via authenticated Socket Mode WebSocket connection)."

**Confidence:** 🟢 High
