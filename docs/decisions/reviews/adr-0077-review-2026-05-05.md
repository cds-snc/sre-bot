## 2. Evidence Gathering & Convention Validation

### 2.A Scope

This review is narrower than a full ADR-0077 review. The amendment touches only the Standard 1 classification table — two row changes. The classification criteria themselves are unchanged. Evidence gathering focuses on the correctness of the classification changes.

**Questions to resolve:**

1. Is JWKS endpoint the correct single backing service for `IdentityService`?
2. Is Category C with no Protocol the correct classification for `SlackPlatformProvider` user lookup?
3. Does Tier 1 (managed service wrapper) remain the correct delegation tier for `IdentityService`?

### 2.B Classification Correctness

| Question | Finding | Confidence |
|----------|---------|------------|
| JWKS endpoint as sole backing service | Confirmed by ADR-0061 S3 amendment: `resolve_from_jwt` performs local claim extraction only; JWKS-based signature verification is performed upstream by `JWKSManager` before the payload reaches `IdentityService`. The only external dependency is the JWKS endpoint URL (release-phase bound). | 🟢 High |
| Category C for `SlackPlatformProvider` user lookup | ADR-0078 explicitly classifies `SlackPlatformProvider` as concrete with no Protocol (Category C pattern). The `users.info` resolution is platform-specific, never substitutable across platforms, and does not require a service contract. Category C is correct. | 🟢 High |
| Tier 1 for `IdentityService` | Tier 1 = managed service wrapper. The only backing service (`JWKSManager` → JWKS endpoint → Google IDP) is a managed service. The delegation is direct and proportional. Tier 1 classification is unchanged and correct. | 🟢 High |

## 3. Assumptions Challenged

### Assumption 1: The table change reflects the ADR-0061 S3 decision — not a new independent decision

**Claim:** The Standard 1 table is a registration table that records decisions made elsewhere. This row change records the scope narrowing decided in ADR-0061 S3. ADR-0077 Standard 1 has no independent normative authority over what `IdentityService` does — that is ADR-0061's domain.

**Challenge Method:** Review ADR-0077 Standard 1's stated purpose.

**Result:** Standard 1 states it is a "canonical classification table" used to determine Protocol migration priority and delegation tier. It is descriptive of decisions governed by domain ADRs. The change is a correct registration update, not a new decision.

**Confidence:** 🟢 High

---

### Assumption 2: Adding `SlackPlatformProvider (user lookup)` to the Standard 1 table as Category C does not mandate a Protocol

**Claim:** Including a Category C row in the Standard 1 table is informational — it records that `SlackPlatformProvider` user lookup is Category C and explicitly does NOT require a Protocol. It does not introduce a Protocol obligation.

**Challenge Method:** Review Standard 1's rules for Category C. Check whether any rule implies Protocol creation for Category C entries.

**Result:** Standard 1 states: "Category C: Implementation Detail — never exposed to features, no Protocol required." The table column "Protocol Required?" is "No" for Category C entries. Adding the row with "No" explicitly reinforces the no-Protocol rule for this capability.

**Confidence:** 🟢 High

---

### Assumption 3: No other Category A services are affected by this amendment

**Claim:** The amendment touches only the `IdentityService` row and adds one Category C row. All other Category A services (`DirectoryProvider`, `StorageService`, `NotificationService`, etc.) are unaffected.

**Challenge Method:** Review remaining Category A rows in Standard 1 for any dependency on `IdentityService` description.

**Result:** No other Category A row references `IdentityService`. The `IdentityService` Protocol scope change has no cascading effect on other services' classification.

**Confidence:** 🟢 High
