## 3. Assumptions Challenged

### Assumption 3.1: The Structural Asymmetry Claim

> **Stated Norm:** "FastAPI provides a DI container only within the request lifecycle. Hookimpls and background jobs execute outside it. The correct DI mechanism is therefore determined by ownership scope and contract stability, not by code location." (Context section)

- **Underlying Assumption:** FastAPI's `Depends()` resolution is irrevocably scoped to the HTTP request lifecycle; no mechanism exists for using FastAPI DI in non-request contexts.
- **Challenge:** FastAPI does have `BackgroundTasks` (Starlette's `starlette.background.BackgroundTasks`), which are functions run *after* the response is sent but within the request lifecycle boundary. These run with access to the same request-scoped dependencies. Could this invalidate the "background job = no FastAPI DI" claim?
- **Evidence Strength:** ⭐ Strong (well-documented in FastAPI/Starlette docs and Python ASGI ecosystem)
- **Counter-Evidence Found:** Yes — partial. FastAPI `BackgroundTasks` (tasks queued via `background_tasks.add_task()`) run within the ASGI lifecycle and could in principle be called with already-resolved dependencies. However, ADR-0086's "Background job" category explicitly refers to in-process scheduled jobs registered via the `register_background_job` hookspec (ADR-0058), which execute on a timer outside any request. FastAPI `BackgroundTasks` is a distinct mechanism and is not the category covered by Standard 4.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The asymmetry claim is structurally correct for the categories ADR-0086 defines. A minor gap exists: the ADR does not explicitly distinguish between ADR-0058 background jobs (timer-based, no request context) and FastAPI `BackgroundTasks` (post-response, within request context). This ambiguity could cause a developer to incorrectly apply Standard 4 (direct provider calls) to a FastAPI `BackgroundTasks` callback that could instead receive resolved dependencies. This is documented as a non-blocking follow-up action.

---

### Assumption 3.2: Direct Provider Calls as "Canonical" for Hookimpls

> **Stated Norm:** "Pluggy hookimpl functions obtain infrastructure services by calling provider functions directly inside the function body. This is the canonical pattern, not a workaround." (Standard 3)

- **Underlying Assumption:** Pluggy's design philosophy necessitates direct provider calls because pluggy has no mechanism to inject arbitrary services into hookimpl bodies.
- **Challenge:** Could one argue that all required services should be passed as hookspec arguments, making direct provider calls unnecessary? If pytest can use a fixture mechanism, why can't this host do the same?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** Yes — partial. The ADR addresses this directly in Alternatives Considered: "Pass all services to hookspecs as arguments — Hookspec signatures become unwieldy (10+ arguments). Breaks pluggy's opt-in argument model. No mature pluggy project (pytest, tox, kedro) uses this pattern." This is accurate. pytest uses a *separate* fixture mechanism (not hookspec arguments) for service injection into test functions — this is not a general pluggy hookspec pattern but a pytest-specific extension. For a general pluggy host, there is no equivalent fixture injection mechanism; the only options are hookspec arguments or internal provider calls.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** S3.2 correctly handles the boundary condition: "If the hookspec signature already provides a dependency as an argument, the hookimpl must use the argument — not re-obtain it via provider call." This is the correct pluggy idiom. The ADR correctly identifies that the choice is between hookspec args (for services the host controls) and direct calls (for everything else), and provides the right rule for each.

---

### Assumption 3.3: Module-Level Prohibition Is Universally Correct

> **Stated Norm:** "No module in `app/packages/` or `app/api/` may call a provider function at module level (outside a function body)." (Standard 5)

- **Underlying Assumption:** No valid use case exists for calling a provider function at module level; the startup lifecycle always must complete before provider functions are called.
- **Challenge:** What about module-level constants derived from static configuration? For example, a module-level `CHUNK_SIZE = 1000` derived from a settings value, where the settings are purely environment variables with no I/O?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No. Even for purely environment-variable-backed settings, the `@lru_cache` function must be called to populate the cache, and this occurs at import time rather than during the startup lifecycle phases defined in ADR-0046. If the module is imported during test setup with a different environment than production, the cached value persists. Additionally, S5.2 specifically addresses this: "Module-level constants derived from settings (`SOME_VALUE = get_settings().key`) are prohibited." The rationale is sound: even if no I/O is required, import-time execution makes the setting invisible to test fixtures, breaks `@lru_cache` teardown patterns, and violates ADR-0048 Boundary 4.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The prohibition is correctly stated and correctly scoped. The stated rationale (import-time execution bypasses startup ordering, per ADR-0046) is accurate.

---

### Assumption 3.4: Consumer-Owned DI Aliases Eliminate the Service Locator Without Breaking the Injection Boundary

> **Stated Norm:** "DI aliases are owned by the consumer, not by infrastructure. A feature package that needs `EventDispatcherDep` declares it in its own `interactions/http.py` or in a package-local `dependencies.py`. There is no global registry of all DI aliases." (Standard 6)

- **Underlying Assumption:** Consumer-owned aliases do not violate ADR-0048 B2 ("all infrastructure services must be consumed through a single, defined injection boundary"), because B2 governs the *consumption surface* (importing from `infrastructure.services`), not the *alias declaration location*.
- **Challenge:** ADR-0048 B2 states "Application code imports only from this boundary." Does declaring an alias in `interactions/http.py` mean the consumer is importing from infrastructure internals rather than from the canonical injection boundary?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No contradiction found. The alias `Annotated[Protocol, Depends(get_x)]` must still import `get_x` from a module in `infrastructure.services.*` — the import path is deep (per ADR-0085 S2.1) rather than from the barrel, but still from within the infrastructure boundary. B2 governs which *module* is imported from, not where the alias declaration *lives*. The alias is syntactic wrapper; the underlying callable reference is still an infrastructure provider function. The distinction is valid and well-reasoned.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** However, one genuine gap exists in the supersession chain: ADR-0056 Standard 4 Rule C1 states "Every `@lru_cache` provider in `providers.py` must have a corresponding `Annotated` alias in `dependencies.py`." ADR-0086 S6.3 deprecates `dependencies.py` per ADR-0085 S3. ADR-0085's supersession metadata explicitly calls out ADR-0056 Standard 4 rules C2 and C3, but does **not** list C1. This creates an unresolved supersession gap: C1 technically still requires aliases in `dependencies.py`, while ADR-0086 S6.3 + ADR-0085 S3 jointly dissolve it. The intent is clear, but the supersession chain is incomplete. See Section 5 and Section 8.

## 4. Failure Modes Identified

### Failure Mode 4.1: Developer Applies Standard 4 to FastAPI `BackgroundTasks`

- **If Assumption Fails:** A developer, following Standard 4 (direct provider calls for "background jobs"), writes a FastAPI `BackgroundTasks` callback using direct provider calls. If the callback runs after the HTTP response has been sent and the per-request DI context has been torn down, this may be correct. However, if the developer wants the callback to use already-resolved, request-scoped dependencies, they would miss the pattern of passing resolved dependencies as closures.
- **Platform Impact:**
  - Incident management workflow: Low
  - Access synchronization workflow: Low (access sync background jobs are explicitly ADR-0058, timer-based)
  - Access request workflow: Low
  - Multi-provider integrations (Slack, Teams, GWS, AWS, GitHub): Low
- **Probability Estimate:** Low (10-15%) — the platform's background jobs are exclusively ADR-0058 timer-based; FastAPI `BackgroundTasks` are not widely used.
- **Mitigation or Acceptance:** The ADR should document that the "Background job" category refers specifically to ADR-0058 `register_background_job` hookspec jobs. FastAPI `BackgroundTasks` are out of scope for Standard 4 and should use already-resolved dependencies passed as closure arguments. Non-blocking clarification.

## 7. Tradeoffs Accepted

### Tradeoff 7.1: Two Mental Models for DI (Depends vs. Direct Call)

- **Chosen:** Context-specific mechanism — `Depends()` for HTTP, direct call for hookimpl/jobs
- **Rejected:** Uniform `Depends()` everywhere (technically impossible outside request context); uniform direct calls everywhere (abandons FastAPI DI, hides dependencies in routes)
- **Rationale:** The asymmetry is structural and permanent — FastAPI's DI resolver is request-scoped by design. Forcing a single pattern where no single pattern is possible would require building a custom DI container. The two-mental-model cost is lower than the custom DI container cost.
- **Risk Accepted:** Developers new to the codebase must learn two resolution patterns. The single-question test ("Am I in a FastAPI route handler?") mitigates but does not eliminate the cognitive cost.
- **Contingency:** The single-question test in Standard 1 and the anti-pattern catalog in Standard 7 are the primary mitigations. If confusion remains persistent, linting rules (Standard 7 enforcement) can make incorrect usage detectable before code review.

---

### Tradeoff 7.2: Consumer-Owned DI Aliases vs. Centralized Aliases

- **Chosen:** Consumer-owned — each consuming file declares its own `Annotated[Protocol, Depends(get_x)]` type alias
- **Rejected:** Centralized `infrastructure/services/dependencies.py` as the authoritative alias registry
- **Rationale:** Centralized aliases exhibit the Service Locator antipattern (any code can import any service alias without declaring ownership). Consumer-owned aliases are self-describing and eliminate speculative declarations. The same alias may appear in multiple files — this is intentional and acceptable.
- **Risk Accepted:** Alias declarations may drift (e.g., inconsistent naming conventions) across consuming files with no enforcement mechanism.
- **Contingency:** Linting rules that verify `Annotated` alias conventions in consuming files. Code review as primary enforcement until lint rules are implemented.

---

### Tradeoff 7.3: `@lru_cache` Direct Provider Calls for Hookimpl Testing (cache_clear Pattern)

- **Chosen:** `get_x.cache_clear()` + monkeypatch as the hookimpl/job test mechanism
- **Rejected:** `dependency_overrides` (not applicable outside FastAPI request context); building a separate DI container for hookimpl testing
- **Rationale:** `@lru_cache` provides its own clear mechanism. `cache_clear()` resets the singleton, allowing monkeypatching of the provider function to return a test double. This is the pragmatic solution within the existing `@lru_cache` provider model.
- **Risk Accepted:** `cache_clear()` must be called in teardown; forgetting it causes test state leakage. Tests that forget teardown can produce false positives or negatives for subsequent tests.
- **Contingency:** ADR-0062 (Testing standards) governs teardown patterns. `@pytest.fixture` with `autouse=True` teardown on `cache_clear` can be established as the package-level test pattern.
