## 3. Assumptions Challenged

Carried forward from R1. Re-evaluated Assumption 3.3 based on revision.

### Assumption 3.3 (Revised): Single Connection per Process Is Sufficient

- **Stated Norm (revised):** "Only one Socket Mode connection exists per process. [...] The application runs multiple ECS tasks (`desired_count = 2`), so Slack distributes inbound events across the resulting connections automatically with no duplication. Horizontal scaling (increasing ECS task count, up to 10) is the preferred throughput strategy; multi-connection per process is not planned."
- **Underlying Assumption:** Event throughput will not exceed capacity of N single-connection tasks (currently 2), and ECS horizontal scaling is sufficient.
- **Challenge:** If throughput requires >10 connections (10 tasks × 1 connection each), the model caps out.
- **Evidence Strength:** ⭐ Strong — current SRE-team operational volume is well within single-connection capacity. 10-task ceiling provides 5× headroom from current 2-task deployment.
- **Counter-Evidence Found:** No — no scenario requires >10 connections for an internal SRE bot.
- **Confidence (ADR survives challenge):** 🟢 High (upgraded from R1 Moderate — revision explicitly documents the scaling model)
- **Reviewer Notes:** The revision closes the R1 gap. Horizontal scaling via ECS task count is documented as the throughput strategy. Multi-connection per process is explicitly marked as not planned. The 10-connection Slack ceiling is acknowledged.

All other R1 assumptions (3.1, 3.2, 3.4, 3.5) remain 🟢 High confidence, unchanged.

## 7. Tradeoffs Accepted

### Tradeoff 7.1: Fail-Fast vs. Degraded-Start for Slack

- **Chosen:** Fail-fast startup when `SLACK_ENABLED=true` but credentials invalid (S1 R3).
- **Rejected:** Degraded-start (run HTTP-only with Slack disabled at runtime).
- **Rationale:** Slack is the primary interaction transport. Running without it in production masks configuration errors.
- **Risk Accepted:** A misconfigured staging deploy will fail to start. Operators must set `SLACK_ENABLED=false` explicitly.
- **Contingency:** `SLACK_ENABLED=false` is the escape hatch. Zero-code-change mitigation.

### Tradeoff 7.2: Single Connection per Process + Horizontal Scaling vs. Multi-Connection per Process

- **Chosen:** Single Socket Mode connection per process; horizontal scaling via ECS task count (S2 R4).
- **Rejected:** Multiple connections per process for load balancing and graceful connection refresh.
- **Rationale:** Operational simplicity — one thread, one handler, deterministic shutdown. ECS horizontal scaling provides the same multi-connection benefits without per-process complexity. Current deployment (`desired_count=2`) already gives 2 connections with automatic event distribution.
- **Risk Accepted:** Max 10 connections via 10 ECS tasks. Per-process multi-connection would allow >10 tasks with >1 connection each, but this exceeds any foreseeable requirement.
- **Contingency:** Multi-connection per process can be added as a provider enhancement without ADR revision.

### Tradeoff 7.3: SDK Mediation vs. Direct Bolt Access

- **Chosen:** Features access Slack exclusively through `SlackPlatformProvider`; no direct Bolt SDK imports (S4 R5).
- **Rejected:** Allow features to import and use Bolt SDK types directly.
- **Rationale:** SDK version isolation — provider absorbs breaking changes. Features depend on stable provider API.
- **Risk Accepted:** Provider API may lag behind Bolt SDK capabilities.
- **Contingency:** Provider API can be extended. `provider.app` property provides escape hatch for advanced Bolt usage.
