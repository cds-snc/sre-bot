### 2.D Validation Summary

**Total Standards Checked:** 10
**Aligned with Best Practice:** 8
**Deliberate Deviations:** 2

**High-Level Finding:**

- 🟡 **Mostly Grounded:** Most standards checked; deviations have rationale but one requires ADR revision.

**Deviation Summary:**

1. **blinker exception propagation vs. error isolation promise** — blinker's `Signal.send()` propagates the first handler exception and stops calling remaining handlers. ADR-0083 Standard 4 point 3 promises error isolation (handler exceptions logged but don't prevent other handlers from executing). The facade cannot simply call `signal.send()` and achieve this guarantee. **Resolution required:** The ADR must prescribe that the facade iterates `signal.receivers_for(sender)` and calls each receiver individually with try/except, or uses a custom wrapper. This is a **blocking** issue because without the prescribed pattern, an implementer could naively use `signal.send()` and break the error isolation contract.

2. **blinker `send()` return type mismatch** — blinker returns `list[tuple[Callable, Any]]` but the current `dispatch_event()` returns `List[Any]`. This is a **non-blocking** implementation detail — the facade adapts the return type. But it should be noted in the ADR for implementer awareness.

## 3. Assumptions Challenged

### Assumption 3.1: blinker is the right library choice

- **Stated Norm:** "Adopt `blinker` as the industry-standard library (Tier 2) for in-process event dispatch" (Decision section)
- **Underlying Assumption:** blinker is the best-fit Python in-process signaling library for this application's constraints.
- **Challenge:** Are there libraries with better error isolation, async support, or typing? PyPubSub provides topic-based messaging with `pub.subscribe()` / `pub.sendMessage()`. pymitter provides namespace-based events. Neither has blinker's ecosystem pedigree.
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No — blinker is MIT licensed, maintained by Pallets ecosystem (Flask maintainers), supports Python 3.9+, is thread-safe, has been actively maintained since 2010 (version 0.8), latest release 2024-11-08 (v1.9.0). It is already used by Flask and Sentry. PyPubSub has weaker maintenance cadence. pymitter is less established.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** blinker is the correct choice. The Pallets ecosystem alignment reduces supply-chain risk. The minimal API surface (Signal/connect/send) matches the application's needs.

### Assumption 3.2: Pluggy for registration timing, blinker for dispatch

- **Stated Norm:** "The hookspec governs *when* handlers register; the library governs *how* dispatch works" (Context, Constraints)
- **Underlying Assumption:** pluggy and blinker serve complementary, non-overlapping roles without redundancy.
- **Challenge:** Could pluggy itself be used for dispatch? pluggy's hook call loop already collects results from all registered hookimpls. Why not use pluggy for both registration *and* dispatch, eliminating blinker entirely?
- **Evidence Strength:** ⭐⭐ Moderate
- **Counter-Evidence Found:** Partial — pluggy could technically dispatch events by defining a `handle_event(event: Event)` hookspec and calling `pm.hook.handle_event(event=event)`. However: (1) pluggy hookimpls don't support filtering by event type natively — all hookimpls would be called for every event, requiring each to filter internally. (2) pluggy's exception handling stops at first failure unless a wrapper catches exceptions. (3) pluggy hooks are LIFO by default with `tryfirst/trylast` modifiers, not the flexible per-signal connection that blinker provides. (4) pluggy is designed for extension points (static hookspecs), not runtime signal dispatch (dynamic connection/disconnection).
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The separation is architecturally sound. pluggy is a plugin registration framework; blinker is a signal dispatch library. Using pluggy for dispatch would conflate plugin discovery with event routing. The ADR correctly identifies that pluggy governs *when* (startup phase 4) and blinker governs *how* (signal routing).

### Assumption 3.3: blinker provides error isolation

- **Stated Norm:** "Handler exceptions during synchronous or background dispatch are caught, logged with structured context [...] and do not prevent other handlers from executing." (Standard 4, point 3)
- **Underlying Assumption:** blinker's dispatch mechanism supports per-handler error isolation.
- **Challenge:** blinker's official documentation explicitly states: "If a receiver raises an exception, that exception will propagate up. This makes debugging straightforward, with an assumption that correctly implemented receivers will not raise." This means `signal.send()` does NOT catch exceptions. The first handler that raises will prevent all subsequent handlers from being called.
- **Evidence Strength:** ⭐ Strong (direct from blinker docs)
- **Counter-Evidence Found:** **Yes** — blinker does NOT provide error isolation. The ADR's Standard 4.3 guarantee cannot be fulfilled by calling `signal.send()` directly. The facade must implement its own error-catching loop.
- **Confidence (ADR survives challenge):** 🔴 Low — **The ADR must be revised to prescribe the error isolation implementation pattern.** Without this, an implementer could naively call `signal.send()` and break the error isolation contract.
- **Reviewer Notes:** This is the single blocking finding. The fix is straightforward: the ADR should specify that `EventDispatcher.dispatch()` must NOT call `signal.send()` directly. Instead, it should iterate `signal.receivers_for(sender)` and call each receiver individually with try/except. Alternatively, it could define a dispatch helper that wraps each handler call. The existing custom `dispatch_event()` function (lines 78-89 of `dispatcher.py`) already implements this pattern correctly — the facade just needs to replicate it using blinker's receiver enumeration API.

### Assumption 3.4: Weak references are safe for handler lifetime

- **Stated Norm:** Handlers register via `signal.connect()` during startup (Standard 3)
- **Underlying Assumption:** blinker's default `weak=True` parameter won't cause handlers to be garbage collected prematurely.
- **Challenge:** If a handler is defined inside a function (e.g., inside a pluggy hookimpl function body) and connected with `weak=True` (default), the handler will be garbage collected when the function scope exits, silently disconnecting it.
- **Evidence Strength:** ⭐⭐ Moderate
- **Counter-Evidence Found:** Unlikely in practice — ADR-0083 Standard 3 requires handlers to be module-level functions registered via pluggy hookspecs. Module-level functions persist for the process lifetime. However, the ADR should note this constraint explicitly.
- **Confidence (ADR survives challenge):** 🟡 Moderate
- **Reviewer Notes:** Non-blocking but worth a documentation note. The ADR should add to Standard 3 that handlers must be module-level callables (not closures or locally-defined functions) to ensure weak reference safety, or specify that `signal.connect(handler, weak=False)` should be used if closure handlers are needed.

### Assumption 3.5: EventDispatcher Category B is correct

- **Stated Norm:** "EventDispatcher remains Category B (shared utility, concrete OK)" (Context, Constraints)
- **Underlying Assumption:** The EventDispatcher does not need a Protocol interface because the underlying library is an implementation detail and will not be swapped.
- **Challenge:** If blinker were later replaced (e.g., with a managed service for cross-process events per ADR-0079 Phase 2), wouldn't a Protocol be needed?
- **Evidence Strength:** ⭐ Strong
- **Counter-Evidence Found:** No — ADR-0077 already addresses this: "Reclassify to Category A if migrated to a durable broker (ADR-0079 Phase 2)." The current B classification is correct for an in-process utility. The reclassification trigger is explicitly documented.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** Well-designed progression path. Category B now → Category A when/if cross-process.

## 4. Failure Modes Identified

### Failure Mode 4.1: Naive `signal.send()` breaks error isolation

- **If Assumption Fails:** An implementer uses `signal.send(event)` directly in the EventDispatcher facade. The first handler that raises an exception terminates dispatch for all remaining handlers. Side effects (notifications, audit logging) are silently lost.
- **Platform Impact:**
  - Incident management workflow: Medium — audit event handlers could be silently skipped
  - Access synchronization workflow: Medium — sync completion notifications could be lost
  - Access request workflow: High — approval notifications to users could fail silently if a preceding handler raises
  - Multi-provider integrations: Low — cross-provider events use managed queues, not in-process signals
- **Probability Estimate:** Medium % — without explicit guidance, this is the obvious implementation path
- **Mitigation or Acceptance:** **Blocker — requires ADR revision.** Add implementation guidance to Standard 4 prescribing the receiver iteration pattern with per-handler try/except. Reference blinker's `signal.receivers_for(sender)` API.

### Failure Mode 4.2: Weak reference handler garbage collection

- **If Assumption Fails:** A handler defined inside a hookimpl function body is connected to a signal. After the hookimpl function returns, the handler is garbage collected. Signal dispatch silently skips the handler.
- **Platform Impact:**
  - Incident management workflow: None — handlers are module-level
  - Access synchronization workflow: Low — if a future developer creates a closure handler
  - Access request workflow: Low — same risk profile
  - Multi-provider integrations: None
- **Probability Estimate:** Low % — current convention uses module-level handlers
- **Mitigation or Acceptance:** Accepted as low risk. Add documentation note to Standard 3. If closure handlers are ever needed, use `weak=False`.

## 7. Tradeoffs Accepted

### Tradeoff 7.1: Library Simplicity vs. Native Error Isolation

- **Chosen:** blinker — minimal API, established ecosystem, zero configuration
- **Rejected:** PyPubSub — more features but heavier, less maintained; custom implementation — full control but violates ADR-0045 P7
- **Rationale:** blinker's simplicity and ecosystem pedigree outweigh the lack of native error isolation. Error isolation is a one-time facade concern, not a recurring maintenance burden.
- **Risk Accepted:** The facade must implement error isolation manually, adding ~10 lines of code to the dispatch method. This is a minimal overhead.
- **Contingency:** If blinker's behavior changes or the library is abandoned, the Protocol-less Category B classification means replacement is contained to the facade.

### Tradeoff 7.2: In-Process Only vs. Cross-Process Capability

- **Chosen:** blinker for in-process only; defer cross-process to ADR-0079 Phase 2
- **Rejected:** Adopting a library with built-in cross-process support (e.g., Celery signals, Kombu)
- **Rationale:** Cross-process delivery is fundamentally a different concern requiring a managed queue service (SQS/SNS). Bundling both concerns in one library would violate separation of concerns and introduce unnecessary infrastructure dependencies.
- **Risk Accepted:** Two ECS tasks cannot share events via blinker. Any feature requiring cross-task events must use SQS/SNS.
- **Contingency:** ADR-0079 Phase 2 provides the evolution path. EventDispatcher reclassifies to Category A with a Protocol at that point.

### Tradeoff 7.3: Weak References vs. Explicit Lifetime Management

- **Chosen:** Default weak references (`weak=True`) — handlers auto-disconnect when garbage collected
- **Rejected:** `weak=False` — handlers persist until explicitly disconnected
- **Rationale:** Module-level handlers (required by Standard 3 + ADR-0048 B4) persist for the process lifetime. Weak references add safety against memory leaks if handlers are inadvertently created in local scopes.
- **Risk Accepted:** A handler defined inside a function scope will be silently garbage collected. Mitigated by Standard 3 requiring module-level handlers.
- **Contingency:** Use `weak=False` for any handler that must persist despite being defined in a non-module scope.
