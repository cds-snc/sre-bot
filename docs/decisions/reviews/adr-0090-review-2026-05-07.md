## 3. Assumptions Challenged

For each architectural norm or constraint stated in ADR-0090, identify and challenge the underlying assumption.

### Assumption 3.1: Single `correlation_id` per entity for entire lifecycle

- **Stated Norm:** "One domain entity has **exactly one** `correlation_id` for its entire lifecycle. The identifier does not change when the entity transitions status, crosses channels, or is re-queried." (Standard 1 C1)
- **Underlying Assumption:** Multi-step interactions require a stable, immutable identifier that survives channel transitions and status changes. No per-channel ID shadowing is needed.
- **Challenge:** Could a feature require per-channel tracking IDs for auditability while maintaining unified entity resolution? (E.g., "approval chain track: HTTP req → Slack approval → Teams notification → HTTP poll")
- **Evidence Strength:** ⭐⭐⭐ Strong
- **Counter-Evidence Found:** No. Structured logging captures per-channel event `platform` fields (ADR-0054). Audit trail is feature + entity + event, not feature + entity + channel_local_id. Single `correlation_id` is sufficient.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** Single `correlation_id` matches distributed tracing conventions (Jaeger, OpenTelemetry, W3C Trace Context). No industry precedent for multi-ID shadowing in stateless systems.

### Assumption 3.2: Minting authority belongs exclusively to one channel per feature

- **Stated Norm:** "A `correlation_id` is minted **exactly once**, at the moment a domain entity is created. The channel and component that may create a new entity (and thus mint a new identifier) is a feature-level decision..." (Standard 2 M1)
- **Underlying Assumption:** Designating a single creation channel prevents ID collisions and ensures minting idempotency is implemented (Standard 2 M2 "entity creation is idempotent").
- **Challenge:** What if a feature supports simultaneous initiation from HTTP and Slack? (E.g., two administrators, one via Backstage UI, one via Slack slash command, attempt to create the same logical request within 1 second.) Will "last-writer-wins" in DynamoDB create silent duplicates or merge correctly?
- **Evidence Strength:** ⭐⭐ Moderate
- **Counter-Evidence Found:** Partially. The ADR-0091 (idempotency standard) governs atomic write with conditional check (TransactWriteItems). If two requests race to create with the same business key (not the same `correlation_id`), the second write fails the condition and reads the first write's result. So merge is idempotent, not duplicate. BUT: ADR-0090 Standard 2 M1 does **not** mandate the business key deduplication logic — it only says "feature-level decision" controls which channel creates. This creates a gap: if a feature allows multi-channel creation without implementing business key deduplication, duplicate `correlation_id`s could result. **Resolution:** This is not an ADR-0090 gap; it's an ADR-0089 + ADR-0091 gap (handler design must enforce business key uniqueness at creation time). ADR-0090 is correct to delegate this to feature ADRs (Tier-4). No blocker.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The gap identified above is a feature-level concern. ADR-0090's role is to govern the `correlation_id` lifecycle, not to prescribe creation policy per feature.

### Assumption 3.3: Slack `private_metadata` is a safe carrier for `correlation_id`

- **Stated Norm:** "Slack `private_metadata` is a JSON-serialized string in the view payload. The minimum required schema is `{"v": 1, "cid": "<uuid>"}`. The `v` field is a version marker for schema evolution." (Standard 3 P2)
- **Underlying Assumption:** Slack's request signature verification (HMAC-SHA256, 5-minute replay window) prevents `private_metadata` forgery. Application-level signing is not needed.
- **Challenge:** Could a malicious Slack workspace admin or a compromised signing secret allow forged `private_metadata` values to reach handlers?
- **Evidence Strength:** ⭐⭐⭐ Strong
- **Counter-Evidence Found:** No. Slack documentation explicitly confirms HMAC signature verification; tampering with any field (including `private_metadata`) invalidates the signature. Socket Mode (used by SRE Bot) leverages pre-authenticated WebSocket — equivalent protection at connection level. Application-level signing would be redundant and incompatible with Slack's architecture.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** Slack's request integrity model is authoritative. Application-level re-signing of `private_metadata` would violate Slack's threat model and add unnecessary complexity.

### Assumption 3.4: Teams Adaptive Card data field (not conversation state) is the authoritative carrier

- **Stated Norm:** "Teams: `Action.Submit` data field | `{"correlation_id": "<uuid>", "intent": "..."}` " (Standard 3 P3)
- **Underlying Assumption:** Adaptive Card data persists reliably in Teams client-side and is returned in `task/submit` invoke payloads. This is more reliable than Bot Framework's archived conversation state ("last-write-wins" race condition).
- **Challenge:** Could Adaptive Card data be lost if the user navigates away before submission? Could the Teams client truncate the data field if it exceeds size limits?
- **Evidence Strength:** ⭐⭐⭐ Strong
- **Counter-Evidence Found:** No. Microsoft Teams docs confirm Adaptive Card data field is rendered in the UI (user-facing); truncation would prevent form submission. Teams stores the card on device; navigation away is an app state loss (not data loss — the card is not submitted). Size limit is ~2KB; standard `correlation_id` + intent field is <100 bytes. ADR-0090 does not attempt to support Teams conversation state (which is archived SDK) or other state carriers. **Note:** The successor Microsoft 365 Agents SDK provides upgraded Teams integration but does not change Adaptive Card payload data field behavior — it remains the idiomatic carrier.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** The archival of Bot Framework SDK (Dec 31, 2025) is a hard constraint that makes conversation state untenable. Adaptive Card data is the correct alternative and is well-supported by Teams.

### Assumption 3.5: 24-hour idempotency TTL balances recoverability and storage cost

- **Stated Norm:** "The idempotency record TTL is set to 24 hours from the record write time...Observe-only endpoints (`GET /feature/entities/{correlation_id}`) do not produce or consume idempotency records." (Standard 4 I2, I4)
- **Underlying Assumption:** 24 hours covers SQS re-delivery (5 min), Slack retries (minutes), and manual operator replays (typical day-in-the-life), but is short enough to avoid DynamoDB storage bloat from expired records.
- **Challenge:** What if an organization's compliance policy requires idempotency records to persist for 90 days? What if SQS dead-letter queue processing is delayed beyond 24 hours?
- **Evidence Strength:** ⭐⭐ Moderate
- **Counter-Evidence Found:** Partially. AWS Builders Library does not prescribe a specific TTL; it depends on context. **However,** ADR-0091 (idempotency standard, which follows ADR-0090) will govern the DynamoDB write pattern and should include a rationale for the 24-hour choice or make it configurable. From a pure research standpoint (tmp/research-concurrent-command-idempotency-strategy-2026-05-06.md), 24 hours is a reasonable default for SRE Bot (typical approval workflows are <24hr). **Resolution:** Not an ADR-0090 blocker; ADR-0091 will add implementation detail and justification. ADR-0090 is correct to specify the TTL model at this abstraction level.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** 24-hour TTL is a sensible default for the use cases documented in Wave 9 (access requests, approvals, incident triage). Future Tier-4 feature ADRs can override if needed (e.g., HIPAA compliance requiring 90-day audit trail).

### Assumption 3.6: Read-only `GET /feature/entities/{correlation_id}` endpoints do not need idempotency records

- **Stated Norm:** "Observe-only endpoints (`GET /feature/entities/{correlation_id}`) do not produce or consume idempotency records. Idempotency applies only to mutable operations." (Standard 4 I4)
- **Underlying Assumption:** GET operations are side-effect-free. Retrying a GET does not require idempotency protection because re-reading the same entity yields the same result.
- **Challenge:** Could a feature implement a GET endpoint that mutates state (e.g., "mark entity as viewed")? Would idempotency be needed then?
- **Evidence Strength:** ⭐⭐⭐ Strong
- **Counter-Evidence Found:** No. HTTP standards (RFC 7231) define GET as safe and idempotent by definition. ADR-0063 (API composition) governs route semantics; a "mark as viewed" operation must use PATCH/POST, not GET. If an operation is mutable, it's not GET. ADR-0090's distinction between "observe-only" (GET) and mutable (POST/PUT/PATCH/DELETE) is aligned with HTTP semantics.
- **Confidence (ADR survives challenge):** 🟢 High
- **Reviewer Notes:** This assumption is rooted in HTTP/REST fundamentals. No deviation from standard practice.

## 4. Failure Modes Identified

For each assumption that scored Moderate or Low confidence, identify specific failure scenarios and their impact on the platform.

### Failure Mode 4.1: Multi-channel creation race condition without business key deduplication

**Assumption:** Assumption 3.2 (minting authority, multi-channel initiation)

**Scenario:** Feature allows both HTTP and Slack to create a request. Administrator A issues HTTP POST at t=0. Administrator B issues Slack slash command at t=0.1s (before HTTP result is committed to DynamoDB). Both minting operations complete; two `correlation_id` values are generated.

**Impact:** Two separate domain entities exist for the same logical business intent (e.g., two approval requests for the same user/resource pair). Duplicate approvals, audit trail confusion.

**Mitigation (in ADR-0089/0091, not ADR-0090):** Handler design (ADR-0089 Standard 2) must enforce that `service.py` validates business key uniqueness and returns the existing entity if a duplicate is detected (idempotent create, per ADR-0091 Standard 2). This is not ADR-0090's responsibility.

**Residual Risk:** LOW — delegated to ADR-0089/0091 and feature-level ADRs. ADR-0090 is clear about scope.

### Failure Mode 4.2: DynamoDB TTL variance beyond 24 hours

**Assumption:** Assumption 3.5 (24-hour TTL, idempotency window)

**Scenario:** Idempotency record is written at t=0 with TTL set to t=86400 (24 hours). DynamoDB background process is delayed by operational issue. Record persists until t=172800 (48 hours). Handler receives duplicate request at t=25 hours; idempotency check is stale, duplicate is processed.

**Impact:** One duplicate approval, transaction, or action reaches the domain (violates at-most-once semantics for that window).

**Mitigation (in ADR-0091):** Handler logic must fall back to semantically equivalent response on duplicate detect, not error. This is covered by ADR-0091 Standard 3 (semantically equivalent response). DynamoDB TTL is a "best effort" guarantee per AWS docs; 48-hour variance is within operational tolerance for non-critical workflows.

**Residual Risk:** LOW — DynamoDB TTL delays are documented and expected. Handled by ADR-0091's response equivalence requirement.

### Failure Mode 4.3: Slack `private_metadata` truncation due to user edits

**Assumption:** Assumption 3.3 (`private_metadata` carries full intent schema)

**Scenario:** A user opens a Slack modal with `private_metadata = {"v":1,"cid":"uuid","intent":"request_approval","source":"backstage"}`. The user somehow intercepts and modifies the modal payload before submission (e.g., via browser DevTools on a web Slack client).

**Impact:** If signature verification is bypassed or misconfigured, a forged `private_metadata` could inject a different `correlation_id` or `intent`.

**Mitigation:** Slack's request signature verification (HMAC-SHA256) is mandatory and non-negotiable. Socket Mode (used by SRE Bot) provides equivalent pre-authenticated WebSocket protection. Application-level validation is redundant and adds no security benefit.

**Residual Risk:** VERY LOW — Slack architecture provides orthogonal protection at request boundary. No mitigation needed in ADR-0090.

## 5. Cross-ADR Consistency Check

**Related ADRs:** ADR-0089 (handler architecture), ADR-0091 (idempotency), ADR-0063 (API composition), ADR-0061 (identity), ADR-0054 (logging)

### Consistency with ADR-0089 (Platform Interaction Handler Standard)

- ✅ **Aligned:** ADR-0090 Standard 1 (single `correlation_id` per entity) matches ADR-0089 Standard 8 (stateless handler invariant). Domain entity `status` in DynamoDB is the source of truth (both ADRs).
- ✅ **Aligned:** ADR-0090 Standard 3 (payload carrier contract) aligns with ADR-0089 Standard 7 (normalised intent — handlers receive `correlation_id` via intent object, not direct platform payloads).
- ✅ **Non-conflicting scope:** ADR-0090 focuses on identity and carrier; ADR-0089 focuses on handler architecture and state choreography. No overlap.

### Consistency with ADR-0091 (Handler Reliability and Idempotency Standard)

- ✅ **Aligned:** ADR-0090 Standard 4 (24-hour TTL for idempotency records) pairs with ADR-0091 Standard 1 (idempotency key schema including `correlation_id`). Both use the same entity PK.
- ✅ **Deferred to ADR-0091:** The exact DynamoDB write pattern (TransactWriteItems, conditional checks) is ADR-0091's scope, not ADR-0090's. ADR-0090 specifies the contract; ADR-0091 specifies the implementation.

### Consistency with ADR-0063 (API Composition and Validation)

- ✅ **Aligned:** ADR-0090 Standard 5 endpoint `GET /feature/entities/{correlation_id}` follows ADR-0063 conventions (typed path parameter, response model, RFC 9457 error schema).
- ✅ **Non-conflicting scope:** ADR-0063 governs API structure (routes, error mapping, OpenAPI). ADR-0090 governs entity resolution by identifier. No overlap.

### Consistency with ADR-0061 (Identity and External Integration Contract)

- ✅ **Non-overlapping scope:** ADR-0061 governs who the authenticated user is. ADR-0090 governs what entity is being operated on (by `correlation_id`). Both are independent.

### Consistency with ADR-0054 (Logging)

- ✅ **Aligned:** ADR-0090 payload carrier contract (Standards 3) ensures `correlation_id` is extractable and loggable. ADR-0054 uses `correlation_id` as a structured log field. Consistent.

## 6. Editorial & Clarification Items (Non-Blocking)

1. **Standard 4 — Minting Authority Clarification:** The ADR states "Feature-level decision, documented in the feature's Tier-4 ADR." Consider adding a forward reference pattern: `[See ADR-0096 (Slack Handler Constraints) for access feature minting rules; ADR-0097 (Teams Interaction Integration) for Teams minting rules.]` This will become boilerplate in Tier-4 ADRs.

2. **Standard 4 — DynamoDB TTL Minting Authority:** The ADR delegates TTL minting to ADR-0091. For clarity, consider a note: `[DynamoDB TTL minting logic is specified in ADR-0091 Standard 2 (atomic write invariant). The ADR-0090 rule here is the TTL value (24 hours), not the implementation.]` This clarifies the boundary between the two ADRs.

3. **Standard 5 — Endpoint Naming Consistency:** The ADR uses the pattern `GET /feature/entities/{correlation_id}`. If a feature uses a different entity noun (e.g., `/access/requests/{correlation_id}`), is this ADR standard still applicable? **Clarification needed in feature ADRs**, but ADR-0090 is correct to use generic `{feature}` and `{entity}` placeholders.
