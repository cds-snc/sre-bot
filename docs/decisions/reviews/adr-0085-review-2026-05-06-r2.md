## 3. Assumptions Challenged

### Assumption 3.1: "The `__init__.py` barrel is a Service Locator"

- **Stated Norm:** "The problem is that the three-file ceremony (ADR-0056 Standard 4) caused `__init__.py` to act as a **Service Locator**: any module can obtain any service by importing from a single address without declaring a dependency on the owning package."
- **Underlying Assumption:** That a Python `__init__.py` re-export module is functionally equivalent to Seemann's Service Locator pattern and therefore inherits all its architectural harms.
- **Challenge:** Seemann's Service Locator is a *runtime registration pattern* — a global dictionary that code queries dynamically (`locator.get(IEmailService)`). A Python `__init__.py` re-export is a *static namespace alias*. The coupling risk in a barrel is real (ownership obscured, import cost forced), but the more precise framing is "aggregation of non-owned symbols that obscures dependency ownership," not "Service Locator" in Seemann's sense. Overstating the analogy could drive unnecessary scope expansion.
- **Evidence Strength:** ⭐ Strong — the practical concern is well-evidenced; the Seemann citation is terminologically imprecise but does not affect the decision.
- **Counter-Evidence Found:** No — the harmful effects (import cost, ownership obscurity) are observable in code. The issue is terminological, not substantive.
- **Confidence (ADR survives challenge):** 🟢 High — the ownership violation and import cost are independently sound justifications. The Seemann framing is supporting context.
- **Reviewer Notes:** A minor qualification would pre-empt terminological disputes: "The barrel exhibits the *effect* of a Service Locator — any consumer can obtain any service from a single address — without the runtime-registration mechanism Seemann describes." Non-blocking.

---

### Assumption 3.2: "`from infrastructure.services import hookimpl` triggers loading of boto3, google-auth, and MaxMind SDKs"

- **Stated Norm:** "Every feature package `__init__.py` contains `from infrastructure.services import hookimpl`. This single import executes the full `__init__.py`, which imports `providers.py`, which imports every infrastructure service module — including heavy SDK wrappers (boto3, google-auth, MaxMind)."
- **Underlying Assumption:** That `providers.py` has module-level imports of heavy SDK client classes, causing those SDKs to be loaded when `providers.py` is imported.
- **Challenge:** Validated by code inspection. `providers.py` lines 13–60 include module-level imports of `AWSClients`, `GoogleWorkspaceClients`, `MaxMindClient`, `SlackClientFacade`, `TeamsClientFacade`, `DiscordClientFacade`. These class imports execute when `providers.py` is first loaded, triggering SDK module-level code. **The claim is verified.**

  **Scope precision:** S6.2 requires "lazy initialization inside the `@lru_cache` function body." This correctly prevents *instance construction* at import time but does **not** prevent *class-level imports* (`from infrastructure.clients.maxmind import MaxMindClient` at module scope) from executing. After barrel dissolution, `from infrastructure.services.providers import get_maxmind_client` still loads all of `providers.py`'s module-level imports. The performance win is realized specifically for the `hookimpl`-only import path, not for any path that needs an actual provider.

- **Evidence Strength:** ⭐ Strong — verified by code inspection.
- **Counter-Evidence Found:** No — the claim is accurate. The scoping nuance is a clarification, not a refutation.
- **Confidence (ADR survives challenge):** 🟡 Moderate — S6.1's claim is valid. S6.2 governs constructor calls, not module-level class imports. The full performance isolation benefit requires function-scoped class imports in `providers.py` — not currently required. This gap limits the scope of S6 but does not invalidate S6.1.
- **Reviewer Notes:** S6.2 should distinguish (a) lazy *instance construction* inside `@lru_cache` bodies (already enforced) from (b) *deferred module imports* for heavy client classes inside provider function bodies (not yet required). Documenting this as a follow-up scope item prevents false expectations. Non-blocking for acceptance.

### Assumption 3.3: "The partial supersession of ADR-0056 S3.2 and S4 C2/C3 is correctly evidenced and scoped"

- **Stated Norm (Compliance section):** "ADR-0056 Standard 3.2 required self-contained providers to be re-exported through `infrastructure.services.__init__.py` for ADR-0048 B2 compliance. This ADR's Standard 1 replaces that requirement… The evidence (PEP 8 explicit imports, Google Style Guide §2.2.4, Seemann's Composition Root pattern) supports deep imports over barrel re-exports."
- **Underlying Assumption:** That PEP 8 explicit imports, Google Style Guide §2.2.4, and Seemann's Composition Root distinction constitute sufficient evidence to supersede a recently deliberated and accepted standard (ADR-0056 S4 was explicitly retained with rationale in April 2026).
- **Challenge:** ADR-0056 S4's rationale for retaining the ceremony was: "It provides explicit traceability from HTTP handler → dependency alias → provider function → service constructor." This is a legitimate architectural concern. The question is whether deep imports provide equivalent traceability without the ceremony overhead.

  Evaluation: PEP 8's "Explicit is better than implicit" applies more strongly to deep imports than to barrel re-exports — a deep import from `infrastructure.services.providers` is more explicit about the source module than `from infrastructure.services import get_x`. Google §2.2.4's preference for explicit module imports is fully consistent with S2's deep import mandate. The Composition Root evidence (Seemann, Cosmic Python) validates the preservation of `providers.py` without requiring a re-export barrel. The traceability concern (the ceremony's rationale) is adequately addressed by IDE "go to definition" and the explicit deep import path — both navigate directly to the provider function. **The evidence chain is sufficient.**

  The supersession scope is precise: S3.2 (re-export requirement for self-contained providers) and S4 C2/C3 (re-export rules from `__init__.py`) are superseded; S4 C1 (every provider has a `dependencies.py` alias — dissolving during S3 migration) and S4 C4/C5 (naming conventions) remain applicable during migration. The ADR explicitly lists retained ADR-0056 provisions, which is correct.

- **Evidence Strength:** ⭐ Strong — three independent authoritative sources support the supersession direction.
- **Counter-Evidence Found:** No — no standard contradicts deep imports in favor of barrel aggregation. ADR-0056's ceremony rationale (traceability) is addressed by the deep import mechanics themselves.
- **Confidence (ADR survives challenge):** 🟢 High — the supersession is correctly scoped, evidence-backed, and does not contradict any parent-tier constraint. Evidence supports ADR-0085 over ADR-0056 S3.2/S4 C2/C3; those provisions needed updating, and the supersession is the correct resolution.
- **Reviewer Notes:** The supersession declaration in metadata and body text is now well-formed. Note that ADR-0056's `superseded_by` field has not been updated to reference ADR-0085 — this is an action item (see Section 8).

### Assumption 3.4: "Deep imports from `providers.py` satisfy ADR-0048 Boundary 2"

- **Stated Norm (Compliance section):** "This redefinition narrows ADR-0048 B2 from 'one barrel address' to 'three explicit import categories' — each with clear ownership and purpose."
- **Underlying Assumption:** That explicitly governing three import categories (`infrastructure.services`, `infrastructure.services.providers`, `infrastructure.<domain>.protocol`) constitutes a valid redefinition of ADR-0048 B2's "single injection surface" rather than a violation of it.
- **Challenge:** ADR-0048 B2 states "Application code imports only from this boundary — never from infrastructure internals." `infrastructure.services.providers` is a module internal to the `infrastructure.services` package. The dissolution ADR defines three explicit categories with distinct purposes and clearly stated consumer populations. The question is whether "three governed categories" is a valid evolution of "one surface."

  Evaluation: ADR-0048 B2 does not mandate a single *file* as the injection boundary; it mandates a "single, defined" boundary. The Compliance section's injection surface table now defines the boundary with precision: each import category has a clear owner, a clear purpose, and a clear consumer class. Concrete implementation modules (`infrastructure.<domain>.service`, `infrastructure.<domain>.dynamodb`) remain off-limits. This is a narrowing of what B2's boundary exposes, not an abandonment of the boundary principle. The redefinition is explicit, governed, and traces back to ADR-0048's intent.

  **One gap in the table:** The "Consumer" column for `infrastructure.services.providers` lists "Hookimpls, background jobs, feature providers" but does not mention route handlers. Route handlers that write consumer-owned DI aliases (`Annotated[Protocol, Depends(get_x)]`) also import `get_x` from `infrastructure.services.providers`. This is implicit from S3 context but not explicit in the table.

- **Evidence Strength:** ⭐⭐ Moderate — the table is clear but has a minor consumer-column gap.
- **Counter-Evidence Found:** No — B2 does not require a single file, only a defined surface. The injection surface table satisfies that requirement.
- **Confidence (ADR survives challenge):** 🟢 High — the injection surface definition adequately resolves the prior-review B2 ambiguity. The minor consumer-column gap (route handlers not listed) is a documentation precision issue, non-blocking.
- **Reviewer Notes:** Add "Route handler DI alias declarations" to the consumer column for `infrastructure.services.providers` to close the documentation gap. Non-blocking.

## 4. Failure Modes Identified

### Failure Mode 4.1: S6.2 Scope Gap → Developer Expectation Mismatch on Import Performance

**(Assumption 3.2, Moderate confidence — non-blocking)**

- **If Assumption Fails:** Developers implement S6 believing that barrel dissolution eliminates all heavy SDK loading at import time. They are then surprised that `from infrastructure.services.providers import get_maxmind_client` still loads all of `providers.py`'s module-level imports (`MaxMindClient`, `AWSClients`, etc.). Tests expecting deep imports to reduce load scope will be disappointed.
- **Platform Impact:**
  - Incident management workflow: Low — production runtime unaffected.
  - Access synchronization workflow: Low — full provider import is expected and acceptable.
  - Access request workflow: Low — same.
  - Multi-provider integrations (Slack, Teams, GWS, AWS, GitHub): Low — full provider import is expected.
  - Test suite: Medium — unit tests that import a single provider still pay the full `providers.py` module load, limiting isolation benefit.
- **Probability Estimate:** Medium — likely to affect test authors, not production behavior.
- **Mitigation or Acceptance:** S6.2 should acknowledge that the `hookimpl`-only path is where the full isolation win occurs. Function-scoped class imports inside provider function bodies would extend the benefit to all provider paths but are a follow-up scope item. Documenting this boundary prevents false expectations without blocking acceptance.

### Failure Mode 4.2: Compatibility Re-exports Adopted by Non-Frozen-Zone Code

**(Assumption: S5.3 enforcement efficacy)**

- **If Assumption Fails:** S5.3 states "No non-frozen-zone code may import these compatibility re-exports." Without a lint rule covering S5.1 exports, non-frozen-zone developers may adopt them as a convenience shortcut. **Resolution:** The updated ADR's Enforcement section now explicitly states the lint rule "includes frozen-zone compatibility re-exports (S5.1) used by non-frozen-zone code." The enforcement mechanism is specified.
- **Platform Impact:**
  - All workflows: Low if enforcement is implemented; Medium over time if enforcement is absent.
- **Probability Estimate:** Low, given the lint rule is now explicitly scoped.
- **Mitigation or Acceptance:** Lint rule implementation must be tracked as a deliverable. The ADR's enforcement specification is sufficient.

## 7. Tradeoffs Accepted

### Tradeoff 7.1: Import Explicitness vs. Consumption Ergonomics

- **Chosen:** Deep imports from defining modules (`from infrastructure.services.providers import get_x`), trading ergonomic brevity for explicit ownership and traceable dependency declarations.
- **Rejected:** Barrel retention with lazy imports, partial dissolution, or `__init__.py` re-exports of all providers.
- **Rationale:** The barrel's import cost affects the plugin registration path (critical startup performance). Ownership violation creates governance friction and obscures where services are defined. PEP 8 and Google Style Guide recommend explicit imports. The verbosity cost is real but bounded.
- **Risk Accepted:** Deep import paths couple consumers to `infrastructure/services/` internal file structure. If `providers.py` is restructured in the future, all deep import consumers break.
- **Contingency:** S4 placement criteria include "change isolation — moving must not increase blast radius." Any `providers.py` restructuring requires a migration plan per S4.4. Risk is accepted and governed.

---

### Tradeoff 7.2: Frozen Zone Backward Compatibility vs. Clean Dissolution

- **Chosen:** Retain compatibility re-exports in `__init__.py` for `app/modules/` frozen zone, marked `# COMPAT`.
- **Rejected:** Immediate dissolution including frozen zone breakage.
- **Rationale:** The frozen zone is a zero-touch policy area. Breaking it to achieve architectural purity creates disproportionate risk for negligible benefit. Compatibility re-exports with markers provide a clear deprecation signal.
- **Risk Accepted:** Compatibility period may extend indefinitely if frozen zone thaw is deprioritized. Without lint enforcement, non-frozen-zone code may adopt compat re-exports.
- **Contingency:** S5.3 enforcement via lint rule, now explicitly scoped in the Enforcement section to include S5.1 compat re-exports.

---

### Tradeoff 7.3: `dependencies.py` Dissolution vs. DI Alias Traceability

- **Chosen:** Dissolve `dependencies.py`; consumer-owned DI aliases per ADR-0086 S1.4.
- **Rejected:** Retain centralized `dependencies.py` ceremony per ADR-0056 S4.
- **Rationale:** Centralized alias declarations force every HTTP handler to import from `dependencies.py`, which triggers the full barrel load. Consumer-owned aliases import only the specific provider function needed. ADR-0056 S4 C2/C3 is now superseded with evidence (PEP 8 explicit imports, Google §2.2.4, Seemann Composition Root pattern).
- **Risk Accepted:** Loss of single-location alias inventory. Discoverability requires IDE tooling or grep rather than one file.
- **Contingency:** IDE "go to definition" and grep patterns for `Annotated[..., Depends(...)]` address discoverability. The supersession of ADR-0056 S4 C2/C3 is now declared and evidenced.
